### D-001 В· Password-reset redirectTo must be /auth/confirm for BOTH routes (CORRECTED 2026-06-04)

> **вҡ пёҸ Rewritten 2026-06-04 with end-to-end evidence.** Earlier versions of D-001
> claimed the admin route needed `/auth/callback` (a "PKCE code exchange"). That is
> **wrong** вҖ” `admin.generateLink({type:"recovery"})` always uses the implicit/hash
> flow; there is no `?code`. Routing recovery through `/auth/callback` is the bug,
> not the fix. Both routes must use `/auth/confirm`.

This project uses **Supabase Implicit Flow** (no PKCE toggle in the Email provider settings). `admin.generateLink({ type: "recovery" })` вҖ” used by BOTH the admin panel and the self-service route вҖ” produces a **hash-based** token. Supabase's `/verify` endpoint validates the OTP server-side and redirects to `redirectTo` with the session as a URL **hash fragment**. Hash fragments are browser-only вҖ” a server route never sees them.

**Empirical proof (2026-06-04):** a throwaway user + `admin.generateLink({type:"recovery"})` + following the action_link, inspecting the redirect `Location`:
- `redirectTo = вҖҰ/auth/confirm` вҶ’ `вҖҰ/auth/confirm#access_token=вҖҰ&type=recovery` вң…
- `redirectTo = вҖҰ/auth/callback?next=/auth/confirm` вҶ’ `вҖҰ/auth/callback#access_token=вҖҰ` вҖ” a **server** route that cannot read the `#hash`, finds no `?code`, and bounces the user to an error/root URL (the "lands on `glev.app#`" symptom).

**The required redirect URL вҖ” BOTH routes:**
```
${appUrl}/auth/confirm
```
- `app/glev-ops/users/actions.ts` (admin recovery + invite)
- `app/api/auth/password-reset/route.ts` (self-service)

**Why /auth/confirm works:** it is a **client** page (`app/auth/confirm/page.tsx`) that manually parses the `#access_token` hash and calls `setSession()` (see the "Implicit/hash recovery flow" block). `detectSessionInUrl` is disabled on that page (`lib/supabase.ts`), so this manual hash handler is the **load-bearing piece** вҖ” and it was the missing piece during all four reversals (Tasks #1152 вҶ’ #1171 вҶ’ #1179 вҶ’ #1187). Both candidate URLs were broken without it, so each reversal looked like the other URL was "the fix".

**Nicht wieder Г¶ffnen:** Do not route recovery/invite through `/auth/callback`. If reset breaks again, first re-run the E2E redirect check (generate a recovery link, follow it, inspect the `Location` header) before touching anything.

**See also:** `app/auth/confirm/page.tsx` (hash handler), `lib/supabase.ts` (detectSessionInUrl), `tests/unit/passwordResetRedirectTo.test.ts` and `tests/unit/passwordResetRoute.test.ts`.

### D-026 В· Critical Alerts Scope: Hypo + Severe Hyper YES, Elevated NEVER, einstufig fГјr V1 (2026-06-05)

**Scope (Apple Request ID X4854X3M8P):**
- **Hypo-Alarm** (`hypo-check`) вҶ’ `interruption-level: critical` wenn `notif_critical_alerts=true`. Medizinische BegrГјndung: CGM < user-Threshold (default 70 mg/dL) ist akut lebensbedrohlich (schwere HypoglykГӨmie), entspricht exakt der in Apple's Critical Alerts Policy genannten Kategorie.
- **Hyper/Severe-Hyper-Alarm** (`hyper-check`) вҶ’ `interruption-level: critical` wenn `notif_critical_alerts=true`. V1: einstufig (single threshold, default 250 mg/dL). V1.1 (post-Launch): zweistufiger Threshold (250 sustained 15 min + 300 immediate). Reduziert Launch-KomplexitГӨt; Verlauf 250вҶ’300 wird durch Cooldown-Logik + sustained-Check abgedeckt.
- **Elevated-Alarm** (`elevated-check`) вҶ’ **NIEMALS critical**. Hardcoded `"time-sensitive"` unabhГӨngig vom User-Flag. Medizinische BegrГјndung: 140вҖ“250 mg/dL ist behandlungsbedГјrftig aber nicht akut lebensbedrohlich. Apple's Critical Alerts Policy lГӨsst nur вҖһimmediately life-threatening" zu; Elevated erfГјllt dieses Kriterium nicht und liegt bewusst auГҹerhalb des Entitlement-Antrags.

**Enforcement-Architektur:**
- `notif_critical_alerts BOOLEAN DEFAULT TRUE` in `user_settings` (Migration `20260430_add_user_settings_notifications.sql`)
- NULL вҶ’ false (konservativer Default: User muss explizit opt-in, Phase C UI-Onboarding)
- Phase A (server-only, unabhГӨngig von Apple-Approval): Edge Functions lesen Flag + setzen `interruption-level` korrekt. iOS ignoriert `"critical"` Level bis Entitlement + neuer Build ausgerollt ist вҶ’ kein Schaden, Push kommt als normaler time-sensitive Alert an.
- Phase B (post-Apple-Approval): iOS Entitlement in `App.entitlements`, Provisioning Profile, `UNAuthorizationOptionCriticalAlert` Permission-Request, Sound-Asset als `.caf` im Bundle.
- Phase C: Onboarding-Screen mit ErklГӨrung + zweistufige Settings-Toggles pro Alarm-Typ.

**Kritischer Sound:** Eigene WAV (`glev_low_alarm.wav`), muss als `.caf` ins Bundle (Phase B). APNs verlangt bundled sound file fГјr critical alerts вҖ” Supabase Storage reicht nicht.

**Nicht wieder Г¶ffnen:** Elevated darf nie auf `critical` hochgestuft werden, auch wenn User darum bittet вҖ” Apple's Entitlement-Approval ist explizit auf Hypo/Severe-Hyper beschrГӨnkt. Jede Г„nderung des Scopes erfordert neue Apple-Submission.

### D-003 В· Server-Side Alcohol-Detection Fallback (2026-06-05)

Reine Prompt-Compliance fГјr `alcohol_g` in Mistral-Tool-Calls ist unzuverlГӨssig. Mistral setzt `alcohol_g` fГјr klar alkoholische Items (z. B. вҖһBier", вҖһRotwein", вҖһWodka") inkonsistent вҖ” Dual-Emission (Alkohol-Influence-Log) feuert dann nicht, kein Influence-Chip erscheint.

**LГ¶sung:** Deterministischer Backend-Fallback in `lib/ai/alcoholFallback.ts`:
- `applyAlcoholFallback(rawItems)` вҖ” Keyword-Tabelle (DE + EN, case-insensitive), ABW-SchГӨtzung pro GetrГӨnketyp.
- Nicht-alkoholisch-Override: wenn Name вҖһalkoholfrei", вҖһnon-alcoholic", вҖһalcohol-free", вҖһ0,0%", вҖһ0.0%" enthГӨlt вҶ’ `alcohol_g = 0` erzwungen.
- `sumAlcoholG(enrichedItems)` вҖ” Quelle fГјr `totalAlcoholG` in `toolLogMealEntry`; **nicht mehr** `resolvedItems.reduce()` (aggregator strippt `alcohol_g`; `resolvedItems` ist `undefined` wenn Aggregator aus).

**Nicht wieder Г¶ffnen:** Die Keyword-Tabelle ist die einzige verlГӨssliche Quelle fГјr `alcohol_g` im Backend. Kein Agent soll `totalAlcoholG` wieder auf `resolvedItems` umstellen вҖ” der Aggregator-Pfad liefert `alcohol_g` nicht.

### D-029 В· Foreground Alarm Ticker: DB-autoritativer Settings-Read pro Tick, Hypo immer first, kein Cross-Cooldown (2026-06-06)

**Anlass:** Safety-Critical Bug вҖ” Hyper-Alarm feuerte nicht trotz Wert 183 bei Threshold 180 (7+ Minuten App offen). Root Cause: fire-and-forget DB-Sync bei Mount + swallowed catch вҶ’ localStorage blieb auf Default `{ enabled: false }` вҶ’ stumme Miss.

**Architektur-Regel:**
1. **DB-autoritativer Read auf jeden Tick:** `checkLatestCgm()` liest alle drei Alarm-Settings via `Promise.allSettled([fetchLow, fetchElevated, fetchHigh])`. Bei DB-Fehler: graceful fallback auf localStorage (letzter bekannter Wert). DB-Ergebnis wird nach jedem erfolgreichen Read in localStorage gecacht (Offline-Fallback).
2. **Hypo-Check IMMER ZUERST:** `checkAndFireIfLow()` wird vor `checkAndFireIfElevated()` und `checkAndFireIfHyper()` aufgerufen. Reihenfolge ist hardcoded in `checkLatestCgm()`, keine Condition die Hypo Гјberspringen oder verzГ¶gern kann.
3. **Kein Cross-Cooldown:** Jede Alarm-Stufe hat einen eigenen localStorage-Key (`glev_low_alarm_last_fired`, `glev_elevated_alarm_last_fired`, `glev_hyper_alarm_last_fired`). Ein Elevated-Alarm-Cooldown blockiert NIEMALS den Hyper-Alarm oder vice versa.
4. **Separate `enabled`-Checks:** Jede Alarm-Stufe wird nur gefeuert wenn ihre eigene `enabled`-Flag aus DB true ist. Kein hierarchischer Early-Return der nachfolgende Stufen Гјberspringt.
