# DECISIONS.md
Dieses Dokument hält bewusste Architektur- und Scope-Entscheidungen fest sowie einen chronologischen Log aller abgeschlossenen Tasks. Zukünftige Agents lesen es am Task-Start, um den Kontext zu verstehen.
---
## Decisions
> Was hier steht, ist eine bewusste Nicht-Entscheidung oder eine Weichenstellung, die nicht mehr offen diskutiert werden soll.
### D-001 · Supabase statt eigener Auth/DB-Infrastruktur (2026-01-15)
Supabase liefert PostgreSQL, Row Level Security und Auth (Email/Password + JWT) als verwalteten Service. Eine eigene Postgres-Instanz mit Auth-Stack hätte dieselbe Betriebslast ohne den Mehrwert. Firebase wurde geprüft, aber wegen SQL-Querybarkeit und Open-Source-Kompatibilität verworfen. **Nicht wieder öffnen:** Solange keine Multi-Tenant-SaaS-Isolation erforderlich ist, bleibt Supabase die Datenebene.
### D-002 · Capacitor statt React Native für mobile Shells (2026-01-20)
Glev ist primär eine Next.js-Web-App. Capacitor erlaubt es, exakt dieselbe Web-Codebasis als Thin-Webview-Shell auf iOS/Android zu verpacken — kein zweiter Rendering-Pfad, keine doppelte Feature-Implementierung. React Native hätte eine separate Komponentenbibliothek und Logik-Schicht erfordert. **Nicht wieder öffnen:** Solange die App keine tief nativen UI-Patterns (z. B. UIKit-Navigation) braucht, bleibt Capacitor die Native-Schicht.
### D-003 · Keine direkten Dosis-Anweisungen in der UI (2026-02-01)
Glev ist ein Entscheidungs-*Support*-System, kein Medizinprodukt Klasse IIb. Direkte Dosis-Anweisungen (z. B. „Nimm jetzt 4 IE") würden eine klinische Validierung und MDR-Zertifizierung erfordern, die den aktuellen Rahmen sprengen. Alle Engine-Empfehlungen sind als Gesprächsbasis fürs Diabetes-Team gerahmt. **Nicht wieder öffnen:** Selbst nach einer möglichen MDR-Einreichung braucht jede Änderung an diesem Prinzip eine explizite Freigabe durch das medizinische Verantwortungsteam.
### D-004 · next-intl statt i18next für Lokalisierung (2026-02-10)
`next-intl` ist für Next.js App Router (Server Components, `generateMetadata`, Middleware-basiertes Locale-Routing) nativ ausgelegt. i18next benötigt zusätzliche Adapter (`next-i18next`) und arbeitet primär Client-seitig, was SSR-Hydration-Mismatches erzeugt. Unterstützte Locales: `de` (Standard) und `en`. **Nicht wieder öffnen:** Eine Migration würde alle `useTranslations()`-Aufrufe und die Middleware betreffen — Aufwand ohne funktionalen Mehrwert.
### D-005 · Pump-Dosierung bewusst aus dem Engine-Scope ausgeschlossen (2026-02-15)
Die Glev Engine berechnet Bolusempfehlungen für ICT-Nutzer (Pen-Injektionen). Insulinpumpen-Nutzer haben eigene Basal-/Bolusprofile, die von der Engine nicht modelliert werden. Die Engine zeigt Pump-Trägern kein Ergebnis an (oder einen expliziten Hinweis). Pump-spezifische Kalkulation (TBR, Extended Bolus) ist kein aktueller Roadmap-Punkt. **Nicht wieder öffnen:** Pump-Support erfordert ein eigenes Datenmodell und klinische Expertise — separates Feature-Flag, wenn überhaupt.
### D-006 · Vercel als einzige Production-Plattform, Replit nur Dev (2026-03-01)
Replit-Secrets fließen nicht in den Vercel-Build. Alle Produktions-Envvars leben in Vercel Project Settings. GitHub Actions (`flush-outbox.yml`) und Stripe-Webhooks zeigen direkt auf `https://glev.app`. Replit wird ausschließlich für Entwicklung und Agent-Tasks genutzt. **Nicht wieder öffnen:** Eine Vermischung der beiden Umgebungen (z. B. Replit Deploy) würde Webhook-Endpunkte und Cron-Jobs duplizieren und Secret-Management verkomplizieren.
### D-010 · ai_consent-Spalte in user_settings — Default false (2026-05-22)
Neue Spalte `ai_consent boolean NOT NULL DEFAULT false` in `user_settings` (Migration `20260522_ai_consent.sql`). Steuert, ob Short-Press auf den Glev AI FAB das `AiHelperSheet` öffnet oder einen "Coming soon"-Toast zeigt. Default false, weil die AI-Features noch kein echtes Backend haben — Nutzer sollen erst explizit zustimmen (UI kommt später). Bestehende Nutzer sehen den Toast; nach explizitem Opt-in öffnet sich das Chat-Sheet.
**Nicht wieder öffnen:** Wenn echtes AI-Backend integriert wird, `DEFAULT` auf `true` setzen oder die Consent-Abfrage ganz entfernen und stattdessen im Sheet einen Disclaimer zeigen.
### D-007 · Engine-Algorithmus: gewichteter k-NN-Ansatz statt LLM-only (2026-05-20)
**Warum kein LLM-only-Approach?**
Ein LLM könnte prinzipiell aus dem Freitext-Meal-Log direkt eine Dosisempfehlung generieren. Das wurde bewusst verworfen:
- **Nicht deterministisch:** Gleiche Eingabe, unterschiedliche Ausgabe — bei klinischen Empfehlungen nicht akzeptabel.
- **Nicht personalisierbar:** Ein generisches LLM kennt nicht das individuelle ICR, die Insulinsensitivität oder das Tageszeit-Muster des Nutzers.
- **Nicht auditierbar:** Der Reasoning-Pfad eines LLM ist für Compliance-Zwecke (MDR) nicht hinreichend dokumentierbar.
- **Latenz & Kosten:** Jede Dosisempfehlung würde einen API-Call erfordern; beim gewichteten Ansatz ist das eine lokale Pure-Funktion.
**Gewählter Ansatz: personalisierter, gewichteter Durchschnitt (k-NN-ähnlich)**
Die Engine lernt den ICR des Nutzers aus finalisierten Mahlzeiten (`carbs / insulin`) und gewichtet schlechtere Outcomes (OVERDOSE, UNDERDOSE → 0.3; SPIKE → 0.7) geringer als GOOD (1.0). Das ist konzeptionell verwandt mit einem gewichteten k-Nearest-Neighbour, aber ohne explizite Feature-Distanz — der Nutzungskontext (Tageszeit, Meal-Type) wird über separate Buckets abgedeckt, nicht über eine Ähnlichkeitsmetrik.
**Was das LLM tut (und was nicht):**
- GPT (`lib/ai/systemPrompt.ts`) parst den Freitext-Meal-Log und klassifiziert die Mahlzeit in einen der vier Typen (FAST_CARBS / HIGH_FAT / HIGH_PROTEIN / BALANCED).
- GPT schlägt **niemals** eine Dosis vor und wertet **niemals** eine Dosis aus — das ist im System-Prompt explizit verboten.
- Die Klassifikation fließt in die Spike-Cutoffs (FAST_CARBS=70, HIGH_FAT=40, HIGH_PROTEIN=50, BALANCED=55 mg/dL) und die Meal-Type-Farb-/Label-Darstellung.
**Vollständige Algorithmus-Dokumentation:** `docs/engine-algorithm.md`
**Nicht wieder öffnen:** Solange Glev kein Klasse-IIb-Medizinprodukt ist und keine klinische Validierung eines LLM-Dosierungsmodells vorliegt, bleibt die deterministische, auditierbare Pure-Function-Pipeline die einzige Dosis-Logik.
### D-009 · Telegram als Agent-Kommunikationskanal (Message-Bus via Supabase) (2026-05-20)
Der Agent (Replit) kann Lucas Fragen stellen und auf Antworten warten, ohne dass Lucas Replit öffnen muss. Der Kanal besteht aus drei Teilen:
1. **Supabase-Tabelle `agent_messages`** — `outbound`-Zeilen werden vom Script geschrieben, `inbound`-Zeilen vom Webhook-Receiver. RLS: service role only (kein anon/authenticated-Zugriff).
2. **`scripts/notify-telegram.mjs`** — ESM-Script, schreibt `outbound`-Zeile, schickt Telegram-Nachricht, wartet via Supabase-Realtime-Subscription auf `inbound`-Antwort (max. 10 Minuten, dann `TIMEOUT`).
3. **`app/api/telegram/webhook/route.ts`** — Next.js-Route, prüft `X-Telegram-Bot-Api-Secret-Token`, transkribiert Voice-Nachrichten via OpenAI Whisper (`whisper-1`), extrahiert `task_id` per Regex aus dem Reply-Kontext, schreibt `inbound`-Zeile.
**Warum Telegram statt Email/SMS?** Niedrige Latenz (Sekunden statt Minuten), native Reply-Thread-Unterstützung (kein neues Compose-Fenster), kein SMS-Provider-Setup. **Warum Supabase als Bus statt direktem Polling der Telegram-API?** Das Script hat keinen öffentlich erreichbaren Port und kann nicht als Webhook-Receiver fungieren. Supabase-Realtime ist the einzige zuverlässige Kanal zwischen dem Webhook (Vercel) und dem wartenden Script (Replit).
**Nicht wieder öffnen:** Solange der Agent auf Replit und die Produktion auf Vercel läuft, ist dieser hybride Ansatz (Webhook auf Vercel empfängt, Supabase-Bus transportiert, Script auf Replit wartet) die richtige Architektur. Eine direkte Polling-Lösung würde den Telegram-Timeout überschreiten und eine Long-Poll-Infrastruktur erfordern.
### D-008 · Engine-Schwellenwerte als exportierte Konstanten mit Doc-Sync-Check (2026-05-20)
Alle klinisch relevanten numerischen Schwellenwerte der Engine-Pipeline werden als `export const` in ihren jeweiligen Quelldateien gepflegt und sind nicht mehr als Inline-Literale vergraben:
- `lib/engine/evaluation.ts`: `HYPO_THRESHOLD`, `SPIKE_CUTOFF_FAST_CARBS`, `SPIKE_CUTOFF_HIGH_FAT`, `SPIKE_CUTOFF_HIGH_PROTEIN`, `SPIKE_CUTOFF_BALANCED` (neu); `SPEED_SPIKE_MGDL_PER_MIN`, `SPEED_SPIKE_STRONG_MGDL_PER_MIN`, `SPIKE_STRONG_MAGNITUDE_MULTIPLIER` waren bereits exportiert.
- `lib/engine/recommendation.ts`: `DEFAULT_ICR`, `DEFAULT_CF`, `DEFAULT_TARGET`, `SAFETY_BG_MIN`, `MAX_DOSE_UNITS` (neu exportiert).
- `lib/engine/adaptiveICR.ts`: `MIN_BUCKET_SAMPLES`, `OUTCOME_WEIGHT` (neu exportiert).
`docs/engine-algorithm.md` enthält jetzt einen maschinenlesbaren **Threshold Index** (letzter Abschnitt) mit Konstantenname, Quelldatei und Wert. `scripts/check-engine-doc-thresholds.mjs` liest diese Tabelle und vergleicht jeden Eintrag gegen die tatsächlichen Werte im TypeScript-Source. Aufruf: `pnpm run check:engine-doc` (Exit 1 bei Abweichung).
**Warum dieser Ansatz statt reiner Inline-Kommentare?** Inline-Kommentare wie "see X in Y" erinnern nur — sie verhindern nicht, dass der Wert still falsch wird. Die maschinenlesbare Tabelle + Script schließt diese Lücke ohne einen vollständigen TypeScript-Build zu erfordern (das Script ist reines Node.js ESM mit Regex-Extraktion).
**Nicht wieder öffnen:** Wenn neue klinische Schwellenwerte zur Engine hinzugefügt werden, müssen sie (a) als `export const` definiert und (b) im Threshold Index eingetragen werden. `pnpm run check:engine-doc` sollte in CI-Pipelines und vor jedem Release-Build laufen.
### D-012 · Automatisierte DB-Migrations via GitHub Actions (2026-05-23)
`.github/workflows/apply-migration.yml` führt `scripts/apply-migration.mjs --all` nach jedem Push auf `main` aus (nur wenn Migrations- oder Script-Dateien geändert wurden). Die Ausführung ist idempotent: eine neue `schema_migrations`-Tabelle in Supabase verfolgt, welche `.sql`-Dateien bereits angewendet wurden. Neue Dateien werden in alphabetischer (timestamp-basierter) Reihenfolge angewendet; bereits eingetragene Dateien werden übersprungen.
**Benötigte GitHub-Repo-Secrets (einmalig einrichten):**
- `SUPABASE_ACCESS_TOKEN` — Supabase Personal Access Token (https://supabase.com/dashboard/account/tokens)
- `NEXT_PUBLIC_SUPABASE_URL` — Supabase-Projekt-URL (gleicher Wert wie in Vercel)
**Trigger-Strategie:** Der Workflow feuert auf jeden Push auf `main` — ohne Pfad-Filter. Push auf `main` ist der Proxy for „Vercel deployed". Die Migration läuft damit nach jedem Deploy; wenn keine neuen Dateien vorliegen, beendet das Script in Sekunden mit „nothing to do" (idempotent). Ein Pfad-Filter würde das Ziel verfehlen: Migrationen müssen auch dann greifen, wenn ein reiner Code-Push (ohne neue `.sql`-Datei) einen Vercel-Deploy auslöst.
**Bootstrap for bestehende Produktions-DBs (einmalig):** Beim ersten Aktivieren auf einer bestehenden Datenbank (mit bereits manuell angewendeten Migrationen) muss `--baseline` einmalig ausgeführt werden — entweder lokal (`pnpm db:migrate:baseline`) oder per GitHub Actions `workflow_dispatch` with `mode = baseline`. Das erstellt die Tracking-Tabelle und markiert alle vorhandenen `.sql`-Dateien als angewendet, OHNE SQL auszuführen. Danach erkennt `--all` nur neue Dateien als pending.
**Warum GitHub Actions statt Vercel post-deploy-Hook?** Vercel bietet keinen nativen post-deploy-Hook für externe SQL-Ausführung. GitHub Actions hat bereits Zugriff auf Repo-Secrets, läuft nach jedem Push auf `main` (zeitgleich mit dem Vercel-Deploy-Trigger) und ist in der bestehenden CI-Infrastruktur integriert.
**Nicht wieder öffnen:** Die `schema_migrations`-Tabelle ist die einzige Migration-Tracking-Quelle. Niemals manuell Zeilen löschen oder die Tabelle droppen — das würde bereits angewendete Migrationen erneut ausführen. Wenn eine Migration rückgängig gemacht werden soll, eine neue `*_rollback.sql`-Datei erstellen.
### D-013 · Glev AI Phase 2 — Consent auf profiles, Mistral als Chat-Provider (2026-05-23)
**Consent-Quelle:** Die UI liest und schreibt ab sofort `profiles.ai_consent_at TIMESTAMPTZ` + `profiles.ai_consent_version TEXT` (beide bereits in Supabase vorhanden). Tap auf "Aktivieren →" im Consent-Modal ruft `POST /api/ai/consent` auf, das `ai_consent_at = NOW()` und `ai_consent_version = 'v1.0'` setzt. Die alte Spalte `user_settings.ai_consent` (boolean, Phase 1) wird **nicht** mehr gelesen; sie bleibt aber in der DB liegen, bis ein späterer Cleanup-Task sie droppt (DROP gehört bewusst nicht in diesen Task, weil eine versehentliche Reaktivierung von Phase-1-Code sonst sofort hart crashen würde — siehe replit.md Deployment-Sektion zum Vercel/Replit-Split).
**Chat-Provider:** Mistral (`mistral-small-latest`, `maxTokens: 300`, `temperature: 0.4`) — bewusst separat vom OpenAI-Client (`lib/ai/openaiClient.ts`), der weiterhin die Food-Parser-Pipeline befeuert. Begründung: zwei sehr unterschiedliche Jobs (strict-JSON-Parser vs. konversationelle Streaming-Antworten), getrennte Prompts (`lib/ai/systemPrompt.ts` vs. `lib/ai/glevChatPrompt.ts`), getrennte Provider-Quotas/Kosten, getrennte Failure-Modes (Mistral-Ausfall darf den Food-Parser nicht killen und umgekehrt).
**Explizite Nicht-Entscheidungen:**
- **Keine Supabase-Persistenz von Chat-Verläufen.** Konversation lebt nur in `sessionStorage` (max. 10 Nachrichten), wird beim Session-Ende automatisch verworfen. Begründung: niedrigster Datenschutz-Footprint solange Glev kein MDR-Produkt ist; Logs müssten sonst als „medizinische Aufzeichnung" behandelt werden. Wenn später Multi-Device-Chat-History gewünscht ist, neue Migration + RLS-Policy.
- **Kein Phase-2-Settings-Toggle zum Widerrufen des Consents.** Wer den Consent zurückziehen will, kann das aktuell nur über einen Support-Request. Ein eigener Toggle würde einen klaren Revoke-Flow + Löschung der `sessionStorage`-History + UI-Strings + Test-Coverage in Settings erfordern — separater Task.
- **Dummy `contextSnapshot`** (`glucoseSummary`, `iobSummary`, `lastMealDescription`) wird vom `useGlevAI`-Hook gesendet, bis ein Follow-up die echten CGM/IOB/Letzte-Mahlzeit-Werte einbindet. Die API validiert nur die Shape, nicht die Inhalte.
**Sicherheit/Compliance:** `POST /api/ai/chat` ist hart durch vier Gates abgesichert: (1) 401 ohne authedClient-User, (2) 403 ohne `profiles.ai_consent_at`, (3) 429 bei >20 Requests/Minute/User (in-memory Map, per-Function-Instance), (4) 503 ohne `MISTRAL_API_KEY`. System-Prompt (`lib/ai/glevChatPrompt.ts`) verbietet explizit Dosis-Empfehlungen und Diagnosen — spiegelt D-003.
**Nicht wieder öffnen:** Der Consent-Mechanismus ist bewusst timestamped + versioned (nicht boolean), damit eine spätere MDR-relevante Änderung des Disclaimer-Texts den User automatisch zu einem neuen Consent zwingen kann (`ai_consent_version` Vergleich gegen aktuelle Konstante). Wenn der Disclaimer geändert wird: Version bumpen, Hook prüft Version-Mismatch und öffnet das Modal erneut. Provider-Wechsel weg von Mistral: nur `lib/ai/mistralClient.ts` + Stream-Loop in `app/api/ai/chat/route.ts` betroffen — Prompt, Consent und Hook bleiben unverändert.
### D-014 · Glev-AI-Rate-Limit in Supabase statt In-Memory-Map (2026-05-23)
Die 20-Requests-pro-Minute-Grenze in `POST /api/ai/chat` lebt jetzt in der neuen Tabelle `public.ai_rate_limit_hits` (`user_id UUID`, `hit_at TIMESTAMPTZ DEFAULT NOW()`, Index auf `(user_id, hit_at DESC)`, RLS an, keine Policies — nur Service-Role-Zugriff). Migration: `supabase/migrations/20260523_ai_rate_limit_hits.sql`. Die alte per-Function-Instance `Map<string, number[]>` ist gestrichen, weil sie auf Vercel weder über parallel laufende Instanzen noch über Cold Starts hinweg konsistent zählte.
**Warum Supabase und nicht Upstash/Redis?** Wir haben Supabase bereits in der Hand (D-001), Service-Role-Client (`lib/supabaseAdmin.ts`) ist überall verkabelt, und 20/min pro User ist keine Hochfrequenz-Last, die einen dedizierten Key-Value-Store rechtfertigt. Upstash würde ein zusätzliches Secret, eine zusätzliche Rechnungsposition und eine zusätzliche Ausfalldomäne bedeuten.
**Fail-open-Politik:** Wenn die Admin-Client-Erzeugung wirft (kein Service-Key) oder das Count/Insert-Query einen Fehler liefert, gibt `isRateLimited` `false` zurück. Begründung: ein DB-Hiccup darf die UI nicht hart zumachen; das Mistral-seitige Konto-Limit bleibt als zweite Schutzschicht.
**Cleanup:** Jeder erfolgreiche Insert feuert opportunistisch ein `DELETE WHERE hit_at < cutoff` für denselben User (best-effort, kein await auf Result). Damit bleibt die Tabelle proportional zur aktiven Userzahl, ohne dass ein separater Cron nötig wäre.
**Nicht wieder öffnen:** Die Werte `RATE_LIMIT_MAX = 20` und `RATE_LIMIT_WINDOW_MS = 60_000` spiegeln das in der 429-Response sichtbare Versprechen ("max 20 requests per minute") wider — Änderungen müssen auch die Error-Copy anpassen. Migration zu Upstash erst, wenn die Tabelle messbar warm wird (≫1 Insert/s/Instance) oder Mistral-Quoten-Konflikte mit Multi-Tenant-Bursts auftreten.

### D-015 · Glev AI Consent-Revoke via Settings-Toggle (2026-05-23)
Ergänzt D-013, das den Revoke-Toggle ursprünglich bewusst nicht enthielt ("Wer den Consent zurückziehen will, kann das aktuell nur über einen Support-Request"). Mit Task #653 ist diese Nicht-Entscheidung **aufgehoben**: Die Settings-Seite hat jetzt einen "Glev AI"-Abschnitt mit On/Off-Toggle. Off → `DELETE /api/ai/consent` nullt `profiles.ai_consent_at` + `ai_consent_version`, der Client löscht zusätzlich den `sessionStorage`-Chat-Verlauf (`glev_ai_history_v1`) und dispatcht `glev:ai-consent-revoked`, damit die Layout-mounted `useGlevAI`-Instanz Sheet/Modal schließt und den In-Memory-Chat verwirft. On → dispatcht `glev:ai-open-consent-modal` und öffnet damit das bestehende Phase-2-Consent-Modal (Disclaimer wird angezeigt; tatsächlicher Consent-Write erfolgt erst beim Tap auf "Aktivieren →"). Damit gilt weiterhin: niemals stillen Consent ohne Disclaimer.
**Sicherheits-/Compliance-Begründung:** Der Revoke-Flow ist eine harte Voraussetzung vor breiterem Rollout (DSGVO Art. 7 Abs. 3 — "Es muss so einfach sein, die Einwilligung zu widerrufen, wie sie zu erteilen"). Da Chat-Inhalte nur in `sessionStorage` leben (D-013), reicht ein clientseitiger `removeItem` + serverseitiger `UPDATE ... SET ai_consent_at = NULL` für den vollständigen Datenlösch-Anspruch — keine Supabase-History-Tabelle zu räumen.
**Nicht wieder öffnen:** Der Toggle bleibt der einzige UI-Pfad zum Widerruf; das Modal bleibt der einzige Pfad zum Consent-Grant. Wenn später Multi-Device-Chat-History eingeführt wird (eigene neue Migration nötig), muss DELETE zusätzlich diese Tabelle leeren.

### D-016 · Glev AI Phase 3 — Function-Calling-Architektur (Tools server-side, READ-only in Task 1) (2026-05-24)
Phase-3-Erweiterung des Mistral-Chats um Function-Calling. Architektur: Mistral-Tool-Loop läuft in `app/api/ai/chat/route.ts` (Phase 1 = `client.chat.complete` mit `tools`/`toolChoice:auto`, max `MAX_TOOL_ROUNDS=2` Iterationen, Tool-Calls werden server-side via `executeGlevTool` aufgelöst und als `role:"tool"`-Messages angehängt; Phase 2 = `client.chat.stream` über den gleichen `messages`-Array für die finale Antwort an den User). Tool-Definitionen + Executor leben in **einer** Datei `lib/ai/glevTools.ts` mit `GLEV_TOOLS`-Array (Mistral-Schema) + `executeGlevTool(name, rawArgs, sb, userId)`-Dispatcher. Task 1 liefert ausschließlich **READ-Tools** (`get_glucose_status`, `get_active_iob`, `get_meal_history`, `get_bolus_history`, `get_appointments`) — alle gegen die authed `sb`-Client-Instanz, RLS-scoped. IOB nutzt bewusst die bestehende `lib/iob.ts`-Logik (`buildDoses` + `calcTotalIOB`) statt eines neuen Pfads — eine Quelle der Wahrheit. CGM nutzt `getHistory(userId)` aus `lib/cgm` (gleicher Pfad wie `/api/cgm/latest`). Migration `20260523_ai_function_calling_schema.sql` ergänzt `meals.bolus_taken_at`, `meals.pre_check_at` und neue Tabelle `meal_timeline_checks (id, user_id, meal_id, check_type, planned_at, confirmed_at, bg_at_check, created_at)` mit vier RLS-Policies (select/insert/update/delete own) — additiv, idempotent, mit `NOTIFY pgrst, 'reload schema'`. Schema-Spec sprach von `entries`-Tabelle; tatsächlich `meals` benutzt.
**Sicherheits-/Compliance-Begründung:** Tools laufen ausschließlich server-side mit der authed Supabase-Client-Instanz — RLS scoped jeden Query auf `auth.uid()::text = user_id`. Rohdaten (CGM-Zeitreihen, vollständige insulin_logs-Rows) verlassen den Server nicht; Executors geben kompakte Aggregate (`{iobUnits, minutesRemaining}`) oder pro Zeile zusammengefasste Objekte (`{units, name, at, note}`) zurück. System-Prompt verbietet explizit: keine Schätzungen, keine Annahmen, nur Tool-Daten als Quelle. Schreib-Operationen (`log_meal_entry`, `log_bolus_entry`, `add_timeline_check`, `create_appointment`) sind bewusst **nicht** in Task 1 — sie kommen in Task 2 mit eigenem D-XXX hinter einem strukturierten UI-Confirmation-Gate (kein Keyword-Matching „ja"/„ok"). Das Compliance-Prinzip aus D-003 bleibt unverletzt: die AI darf Mahlzeiten/Boli vorlesen, aber keine Dosis empfehlen, und Phase 1 schreibt überhaupt nichts.
**Drei explizite Nicht-Entscheidungen:**
1. **Keine WRITE-Tools in Task 1** — Tools, die in `meals`/`insulin_logs`/`appointments`/`meal_timeline_checks` schreiben, sind absichtlich nicht implementiert. Erst nach UI-Confirmation-Gate in Task 2.
2. **Keine Tool-Result-Persistenz** — Tool-Calls und ihre Ergebnisse leben nur im einen Request-Lifecycle. Wer später ein „warum hat die AI das gesagt?"-Audit-Log braucht, muss eine eigene Migration + neuen D-XXX einführen.
3. **Kein Streaming der Tool-Phase** — Phase 1 ist non-streaming (`chat.complete`), erst die finale Antwort wird gestreamt. Das addiert ~1–2 s Latenz wenn Tools gerufen werden; akzeptabel weil ohne Tools Phase 1 in einem Round leer durchläuft und sofort zu Phase 2 springt.
**Nicht wieder öffnen:** Tool-Definitionen und Executors müssen in `lib/ai/glevTools.ts` synchron bleiben (siehe `GlevToolName`-Union). Neue Tools immer mit (a) RLS-Check auf authed `sb`, (b) summarized Output (kein Raw-Time-Series-Pass-Through), (c) Update am System-Prompt, das die neue Tool-Funktion in der „Tools (echte Nutzerdaten):"-Sektion listet. `MAX_TOOL_ROUNDS=2` bleibt — höhere Werte würden Latenz und Token-Kosten linear erhöhen ohne dass realistische Glev-Use-Cases mehr als zwei Tool-Rounds brauchen.

### D-016 · Persistentes User-Memory für den Glev AI Agenten (2026-05-24)
Neue Tabelle `public.ai_user_memory(user_id uuid REFERENCES profiles(user_id) ON DELETE CASCADE, key text, value text, updated_at timestamptz, PK(user_id, key))` — Migration `20260524_ai_user_memory.sql`. RLS an, vier `auth.uid() = user_id`-Policies (select/insert/update/delete own). Neues Function-Calling-Tool `save_user_observation(key, value)` in `lib/ai/glevTools.ts` mit Server-side-Trim + Längen-Limits (key ≤ 64, value ≤ 500) und Upsert via `onConflict: "user_id,key"` — gleicher Key überschreibt den alten Value, keine Doppelzeilen. - Kein UI zum Anzeigen/Löschen/Editieren der Memory-Einträge durch den User (separate Task, falls gewünscht).

### D-017 · WRITE-Tools für Glev AI laufen ausschließlich über UI-Confirmation-Gate (2026-05-24)
Phase-3-Task-2-Architektur: schreibende Function-Calling-Tools (`log_meal_entry`, `log_bolus_entry`, `log_fingerstick`, `add_appointment`) führen **niemals** direkt einen Insert in `meals`/`insulin_logs`/`fingerstick_readings`/`appointments` aus. Stattdessen:

1. Tool-Executor in `lib/ai/glevTools.ts` validiert die vom Modell vorgeschlagenen Args, baut eine kompakte deutsche `summary` und legt eine Zeile in der neuen Tabelle `public.ai_pending_actions` an (Migration `20260524_ai_pending_actions.sql`: `token uuid PK`, `user_id` FK auf `profiles(user_id) ON DELETE CASCADE`, `kind` CHECK auf die vier Tool-Namen, `params jsonb`, `summary text`, `expires_at` TTL 5 min, `used_at` Idempotenz-Guard, RLS an mit vier `auth.uid()=user_id`-Policies).
2. Tool-Result-Form ist `{ pending_action: { token, kind, summary } }` — der `isPendingActionEnvelope`-Guard in `glevTools.ts` markiert das Pattern, `app/api/ai/chat/route.ts` sniffed es im Tool-Loop, schickt das Objekt per separatem SSE-Frame `{pending_action:{…}}` an die UI und reduziert die Tool-Reply Richtung Mistral auf einen `"awaiting_user_confirmation"`-Stub, damit das Modell weder selbst „bestätigt" noch in derselben Runde weitere WRITE-Calls kettet.
3. `lib/useGlevAI.ts` parst den SSE-Frame und hängt einen `PendingAction { token, kind, summary, state }` ans aktive Assistant-Bubble. `components/GlevAIChatSheet.tsx` rendert darunter ein `PendingActionWidget` mit Bestätigen/Abbrechen — Tap auf Bestätigen ruft `POST /api/ai/confirm-action {token}`, das (a) Token-Ownership prüft (defensive 403 zusätzlich zu RLS), (b) Ablauf prüft (410), (c) Idempotenz via `update … is("used_at", null)`-Race-Claim erzwingt (409 für den Zweitklick), (d) den eigentlichen Insert per Switch auf `kind` ausführt.

**Warum DB-Token statt In-Memory-Map?** Eine In-Memory-Lösung würde Vercels Serverless-Funktions-Lifecycle nicht überleben (Cold-Start verliert die Map). Der DB-Token ist ohnehin nötig für RLS-Ownership-Beweis und TTL.

**Warum `used_at`-Update als Idempotenz-Guard statt DELETE?** Wir behalten den Audit-Pfad (welche Aktion welcher User wann bestätigt hat) und gewinnen die Race-Resistenz „nur der erste Klick gewinnt" durch das Conditional-Update auf `is("used_at", null)`. DELETE würde beide Punkte verlieren.

**Warum Cancel ohne Server-Roundtrip?** Cancel ist rein lokal — die DB-Zeile expired ohnehin nach 5 min. Ein Server-Call wäre nur kosmetisch (Storage-Sparen) und würde einen weiteren Netzwerk-Roundtrip mitten in einem Abbrech-Gestus erzwingen.

**Compliance-Prinzip (verstärkt D-003 + Compliance-Backlog #1):** Glev darf nie selbst Dosen vorschlagen oder „autonom" loggen. System-Prompt (`lib/ai/glevChatPrompt.ts`) enthält explizit den Satz „Niemals selbst eine Bolus-Dosis vorschlagen oder eine Mahlzeit ergänzen mit Werten, die der Nutzer nicht genannt hat. Du dokumentierst, was der Nutzer sagt — du berechnest oder rätst nicht." Jeder WRITE braucht einen expliziten User-Tap auf den Bestätigen-Button — eine Mistral-Halluzination kann maximal eine Confirm-Bubble erzeugen, niemals eine DB-Zeile.

**Datum-Kontext für `add_appointment`:** Im `contextPreamble` steht jetzt das heutige Datum in der Geräte-Zeitzone als YYYY-MM-DD (via neuem `todayInTimezone()`-Helper). Mistral rechnet relative Wörter („nächsten Dienstag") auf das absolute Datum um, bevor das Tool gerufen wird. mmol/L für Fingersticks werden vom Modell × 18 in mg/dL umgerechnet (Prompt-Regel).

**Race-Safety im Detail (nach Architect-Review):** Der Claim-Query `update … is("used_at", null).gt("expires_at", now)` bündelt Idempotenz UND Expiry-Check in einer atomaren Operation — schließt das TOCTOU-Fenster zwischen Pre-Check und Claim. Schlägt der nachfolgende Insert in die Zieltabelle fehl (Constraint, Schema-Lag, transient), rollt der `catch`-Block `used_at` zurück auf NULL (conditional auf `eq("used_at", claimAt)` damit ein paralleler erfolgreicher Reset nicht überschrieben wird) — sonst würde jeder Transient-Error den Token permanent verbrennen und „Nochmal versuchen" im UI wäre wirkungslos. Maximal **eine** pending WRITE-Aktion pro Assistant-Turn: rufte das Modell in einer Runde mehrere WRITE-Tools auf, würde die UI nur die letzte sehen (eine `pendingAction` pro Bubble). Der Tool-Loop in `/api/ai/chat` lehnt den zweiten WRITE der gleichen Runde als `{status:"rejected", reason:"only_one_write_action_per_turn"}` ab, damit Mistral entweder textlich darauf hinweist oder im nächsten Turn nachzieht.

**Compliance-Grenzen dieses Gates:** Das Confirmation-Gate verhindert, dass eine Mistral-Halluzination ohne User-Tap eine DB-Zeile erzeugt. Es verhindert NICHT, dass das Modell im Antwort-Text eine konkrete Dosis-Zahl nennt — dafür sind die expliziten Prompt-Regeln („Niemals selbst eine Bolus-Dosis vorschlagen") und der existierende Disclaimer-Footer zuständig. Ein server-seitiger Output-Filter wäre eine eigene MDR-Maßnahme (Compliance-Backlog #1) und nicht Teil dieses Tasks.

**Nicht wieder öffnen:** Jedes weitere WRITE-Tool (z. B. `add_timeline_check`, `update_meal`, `delete_bolus`) muss (a) als neuer `kind`-Wert in die CHECK-Constraint der Migration aufgenommen werden, (b) den `createPendingAction`-Helper nutzen statt direkt zu schreiben, (c) im `executeConfirmedAction`-Switch von `app/api/ai/confirm-action/route.ts` einen Branch erhalten, (d) in `GlevToolName`-Union + System-Prompt-Tool-Liste landen. Der Confirm-Endpoint darf NIE eine schreibende Operation enthalten, die nicht über `ai_pending_actions` autorisiert wurde — kein „Convenience-Bypass". Das Idempotenz-Guard-Pattern (`update … is("used_at", null).gt("expires_at", now) … select` + `catch`-Rollback) ist die einzige zugelassene Form der Token-Einlösung; ein einfaches SELECT-then-UPDATE würde die Doppelklick-Race verlieren, und ein Claim ohne Rollback würde Transient-Errors permanent verbrennen. Die „max 1 pending WRITE per turn"-Regel im Chat-Tool-Loop bleibt — sonst muss erst die UI auf mehrere `pendingAction`-Widgets pro Bubble umgebaut werden.

### D-020 · Server-seitiger Hypo-Push: Supabase Edge Function + Token-Tabelle auf profiles (2026-05-25)
Task #693 implementiert den in D-019 angekündigten Server-Push als eigene Iteration. Architektur-Entscheidungen:

**Token-Speicherung auf `profiles` statt eigener Tabelle:**
`push_token`, `push_platform` und `push_token_updated_at` landen direkt auf `profiles` (via `ALTER TABLE profiles ADD COLUMN IF NOT EXISTS …`), nicht in einer separaten `push_tokens`-Tabelle. Begründung: Ein User hat genau ein aktives Gerät pro Session (Capacitor-App). Eine separate Tabelle wäre Over-Engineering für den aktuellen Scope und würde eine zusätzliche JOIN-Logik in der Edge Function erfordern.

**Cooldown in eigener Tabelle (`hypo_push_cooldown`):**
15-Minuten-Cooldown analog zum localStorage-Cooldown (D-019), aber persistiert in der DB damit er über Serverless-Instanzen und App-Neustarts hinweg gilt. Service-Role-only (kein RLS für Auth-Users). Separate Tabelle statt Spalte auf `profiles`, weil der Cooldown ein technisches Betriebsartefakt ist und nicht zum User-Profil gehört — trennt Concerns sauber.

**Supabase Edge Function statt Vercel Edge/Serverless:**
Die hypo-check-Funktion läuft auf Supabase (`supabase/functions/hypo-check/`), nicht auf Vercel. Begründung: (1) Sie braucht direkten Service-Role-Zugriff auf die DB ohne Network-Hop; (2) der Cron-Schedule (`*/5 * * * *`) ist nativ in Supabase Edge Functions unterstützt (kein GitHub Actions Cron + HTTP-Call nötig); (3) Push-Secrets (FCM-Key, APNs-P8) gehören nicht in Vercel Environment Variables — sie sind ausschließlich für diese eine Funktion relevant.

**Keine PHI in Push-Payloads:**
Der Push-Body enthält nur den rohen mg/dL-Wert und einen generischen Prompt (`„Dein BZ liegt bei [Wert] — prüf dich jetzt."`). Keine Mahlzeit-Daten, keine Dosis-Empfehlungen, keine klinischen Interpretationen. iOS-APNs-Policies untersagen PHI in Push-Payloads explizit (Apple Developer Program License Agreement §3.3.12).

**Drei explizite Nicht-Entscheidungen:**
1. **Kein Snooze via Push-Action** — Notification-Action-Buttons erfordern native Entitlements und eine eigene Handling-Logik im Capacitor-Shell. Snooze bleibt In-App-Feature.
2. **Kein Multi-Device-Support** — `profiles` speichert nur ein Token (das zuletzt registrierte Gerät). Wenn ein User zwei Geräte hat, erhält nur das zuletzt gestartete den Server-Push. Multi-Device wäre eine separate `push_tokens`-Tabelle mit Device-ID.
3. **Kein Retry bei Push-Fehler** — Ein FCM-/APNs-Fehler wird geloggt, aber nicht wiederholt. Die nächste Cron-Iteration (5 Minuten später) probiert es erneut, falls der CGM-Wert noch niedrig ist und der Cooldown abgelaufen ist.

**Nicht wieder öffnen:** Token-Schema und Cooldown-Tabelle sind absichtlich minimal. Multi-Device oder Retry-Queues erfordern einen eigenen D-XXX mit Scope-Abgrenzung durch den Product Owner.

### D-025 · Facebook Conversion API (CAPI): Architektur-Entscheidungen (2026-05-27)

**Kontext:** Werbe-Tracking für Meta-Kampagnen. DSGVO + iOS14+ schränken Browser-seitiges Pixel-Tracking stark ein — Server-seitiges CAPI ist die einzige zuverlässige Alternative.

**Drei Architektur-Entscheidungen:**

1. **Browser-Pixel + Server-CAPI parallel (Deduplication via `event_id`):**
   Das Browser-Pixel (`fbq("track", ...)`) läuft weiterhin als Primärkanal für schnelle Attribution. Server-CAPI sendet denselben Event über die Meta Graph API. Meta dedupliziert via identische `event_id` (UUID) — kein Double-Count. Nur CAPI würde reichen, aber das Pixel liefert `fbp`-Cookie und PageView-Tracking ohne Extra-Code.

2. **DSGVO-Consent-Gate via `window.__consent.marketing`:**
   `CookieBanner.tsx` setzt `window.__consent = { marketing: true|false }` nach Opt-in/Opt-out und persistiert in `localStorage.glev_cookie_consent`. `trackEvent()` in `lib/fb-capi-client.ts` prüft `hasMarketingConsent()` vor jedem Fire — kein Event ohne Einwilligung. Das Banner zeigt sich beim Erstbesuch (600ms Delay für SSR-Flash-Prevention), danach wird der gespeicherte Consent still wiederhergestellt.

3. **Webhook-CAPI ist fire-and-forget — blockiert nie den Stripe-Return:**
   `sendCapiEvent()` in Stripe-Webhooks ist `.catch()`-gewrappt. Meta-API-Fehler sind kein Grund für einen Stripe-Webhook-Retry (502/503 würde Stripe bis zu 3 Tage lang wiederholen, mit jeder Retry ein doppelter DB-Write). Konsequenz: bei Meta-Ausfällen gehen nur Tracking-Events verloren, nie Käufer-Daten.

**Plan-Differenzierung in Stripe-Metadata:**
`/api/checkout/pro` und `/api/checkout/plus` schreiben `plan_name`/`plan_id` in die Stripe-Session-Metadata. Der Pro-Webhook liest diese Felder und schickt sie als `contentName`/`contentIds` an Meta — so erscheinen "Glev Pro" und "Glev+" als separate `Purchase`-Events im Meta Events Manager.

**Nicht wieder öffnen:** Alle CAPI-Calls laufen über `lib/fb-capi-server.ts` (Server) bzw. `lib/fb-capi-client.ts` + `/api/fb-capi` (Browser-Proxy). Keine direkten Graph-API-Calls außerhalb dieser Dateien. Consent-Check nicht umgehen — auch nicht für A/B-Tests oder Drittanbieter.
