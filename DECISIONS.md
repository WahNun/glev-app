### D-001 · Password-reset redirectTo must be /auth/confirm for BOTH routes (CORRECTED 2026-06-04)

> **⚠️ Rewritten 2026-06-04 with end-to-end evidence.** Earlier versions of D-001
> claimed the admin route needed `/auth/callback` (a "PKCE code exchange"). That is
> **wrong** — `admin.generateLink({type:"recovery"})` always uses the implicit/hash
> flow; there is no `?code`. Routing recovery through `/auth/callback` is the bug,
> not the fix. Both routes must use `/auth/confirm`.

This project uses **Supabase Implicit Flow** (no PKCE toggle in the Email provider settings). `admin.generateLink({ type: "recovery" })` — used by BOTH the admin panel and the self-service route — produces a **hash-based** token. Supabase's `/verify` endpoint validates the OTP server-side and redirects to `redirectTo` with the session as a URL **hash fragment**. Hash fragments are browser-only — a server route never sees them.

**Empirical proof (2026-06-04):** a throwaway user + `admin.generateLink({type:"recovery"})` + following the action_link, inspecting the redirect `Location`:
- `redirectTo = …/auth/confirm` → `…/auth/confirm#access_token=…&type=recovery` ✅
- `redirectTo = …/auth/callback?next=/auth/confirm` → `…/auth/callback#access_token=…` — a **server** route that cannot read the `#hash`, finds no `?code`, and bounces the user to an error/root URL (the "lands on `glev.app#`" symptom).

**The required redirect URL — BOTH routes:**
```
${appUrl}/auth/confirm
```
- `app/glev-ops/users/actions.ts` (admin recovery + invite)
- `app/api/auth/password-reset/route.ts` (self-service)

**Why /auth/confirm works:** it is a **client** page (`app/auth/confirm/page.tsx`) that manually parses the `#access_token` hash and calls `setSession()` (see the "Implicit/hash recovery flow" block). `detectSessionInUrl` is disabled on that page (`lib/supabase.ts`), so this manual hash handler is the **load-bearing piece** — and it was the missing piece during all four reversals (Tasks #1152 → #1171 → #1179 → #1187). Both candidate URLs were broken without it, so each reversal looked like the other URL was "the fix".

**Nicht wieder öffnen:** Do not route recovery/invite through `/auth/callback`. If reset breaks again, first re-run the E2E redirect check (generate a recovery link, follow it, inspect the `Location` header) before touching anything.

**See also:** `app/auth/confirm/page.tsx` (hash handler), `lib/supabase.ts` (detectSessionInUrl), `tests/unit/passwordResetRedirectTo.test.ts` and `tests/unit/passwordResetRoute.test.ts`.

### D-026 · Critical Alerts Scope: Hypo + Severe Hyper YES, Elevated NEVER, einstufig für V1 (2026-06-05)

**Scope (Apple Request ID X4854X3M8P):**
- **Hypo-Alarm** (`hypo-check`) → `interruption-level: critical` wenn `notif_critical_alerts=true`. Medizinische Begründung: CGM < user-Threshold (default 70 mg/dL) ist akut lebensbedrohlich (schwere Hypoglykämie), entspricht exakt der in Apple's Critical Alerts Policy genannten Kategorie.
- **Hyper/Severe-Hyper-Alarm** (`hyper-check`) → `interruption-level: critical` wenn `notif_critical_alerts=true`. V1: einstufig (single threshold, default 250 mg/dL). V1.1 (post-Launch): zweistufiger Threshold (250 sustained 15 min + 300 immediate). Reduziert Launch-Komplexität; Verlauf 250→300 wird durch Cooldown-Logik + sustained-Check abgedeckt.
- **Elevated-Alarm** (`elevated-check`) → **NIEMALS critical**. Hardcoded `"time-sensitive"` unabhängig vom User-Flag. Medizinische Begründung: 140–250 mg/dL ist behandlungsbedürftig aber nicht akut lebensbedrohlich. Apple's Critical Alerts Policy lässt nur „immediately life-threatening" zu; Elevated erfüllt dieses Kriterium nicht und liegt bewusst außerhalb des Entitlement-Antrags.

**Enforcement-Architektur:**
- `notif_critical_alerts BOOLEAN DEFAULT TRUE` in `user_settings` (Migration `20260430_add_user_settings_notifications.sql`)
- NULL → false (konservativer Default: User muss explizit opt-in, Phase C UI-Onboarding)
- Phase A (server-only, unabhängig von Apple-Approval): Edge Functions lesen Flag + setzen `interruption-level` korrekt. iOS ignoriert `"critical"` Level bis Entitlement + neuer Build ausgerollt ist → kein Schaden, Push kommt als normaler time-sensitive Alert an.
- Phase B (post-Apple-Approval): iOS Entitlement in `App.entitlements`, Provisioning Profile, `UNAuthorizationOptionCriticalAlert` Permission-Request, Sound-Asset als `.caf` im Bundle.
- Phase C: Onboarding-Screen mit Erklärung + zweistufige Settings-Toggles pro Alarm-Typ.

**Kritischer Sound:** Eigene WAV (`glev_low_alarm.wav`), muss als `.caf` ins Bundle (Phase B). APNs verlangt bundled sound file für critical alerts — Supabase Storage reicht nicht.

**Nicht wieder öffnen:** Elevated darf nie auf `critical` hochgestuft werden, auch wenn User darum bittet — Apple's Entitlement-Approval ist explizit auf Hypo/Severe-Hyper beschränkt. Jede Änderung des Scopes erfordert neue Apple-Submission.

### D-003 · Server-Side Alcohol-Detection Fallback (2026-06-05)

Reine Prompt-Compliance für `alcohol_g` in Mistral-Tool-Calls ist unzuverlässig. Mistral setzt `alcohol_g` für klar alkoholische Items (z. B. „Bier", „Rotwein", „Wodka") inkonsistent — Dual-Emission (Alkohol-Influence-Log) feuert dann nicht, kein Influence-Chip erscheint.

**Lösung:** Deterministischer Backend-Fallback in `lib/ai/alcoholFallback.ts`:
- `applyAlcoholFallback(rawItems)` — Keyword-Tabelle (DE + EN, case-insensitive), ABW-Schätzung pro Getränketyp.
- Nicht-alkoholisch-Override: wenn Name „alkoholfrei", „non-alcoholic", „alcohol-free", „0,0%", „0.0%" enthält → `alcohol_g = 0` erzwungen.
- `sumAlcoholG(enrichedItems)` — Quelle für `totalAlcoholG` in `toolLogMealEntry`; **nicht mehr** `resolvedItems.reduce()` (aggregator strippt `alcohol_g`; `resolvedItems` ist `undefined` wenn Aggregator aus).

**Nicht wieder öffnen:** Die Keyword-Tabelle ist die einzige verlässliche Quelle für `alcohol_g` im Backend. Kein Agent soll `totalAlcoholG` wieder auf `resolvedItems` umstellen — der Aggregator-Pfad liefert `alcohol_g` nicht.

**Nicht wieder öffnen:** Keine neue Route darf `{ error: "some raw string" }` zurückgeben. Immer `errorResponse(code, status)` aus `lib/api/errorResponse.ts` verwenden. `retry_allowed` ist automatisch gesetzt — nicht manuell übersteuern.

### D-030 · glev_ops_users — Supabase-basierte Team-Zugänge für /glev-ops/* (2026-06-06)

**Anlass:** Env-Var-basiertes Marketer-System unterstützt nur einen Marketer. Weitere Teammitglieder (z. B. kocak@aksme.de) brauchen eigene Logins.

**Architektur:**
- Tabelle `public.glev_ops_users` (id, email, password_hash, role, name, must_change_pw, last_login_at). RLS enabled, service-role-only Policy — kein direkter User-Zugriff.
- Passwort-Hashing via Node `crypto.scrypt` (kein externes Package, Salted, 64-Byte-Key).
- Cookie-Format: `team:{userId}:{role}:{hmac}` — HMAC-SHA256 über ADMIN_API_SECRET auf `team-member-v1:{id}:{role}`. Kein DB-Lookup bei Session-Validierung (nur HMAC-Check).
- Login-Prio in `loginAction`: (1) Admin via Env-Vars → (2) Marketer via Env-Vars → (3) Team-Member via Supabase.
- Erster Login: `must_change_pw=true` → Redirect auf `/glev-ops/team/change-password`. Nach PW-Änderung: `must_change_pw=false`, neues Cookie gesetzt.
- Admin verwaltet Team unter `/glev-ops/team` (Hinzufügen, Reset, Löschen).

**Nicht wieder öffnen:** Der Master-Admin-Login via `ADMIN_EMAIL` + `ADMIN_API_SECRET` (Env-Vars) ist vollständig unabhängig von dieser Tabelle und braucht nie darin zu stehen. HMAC-Key für Team-Cookies ist `ADMIN_API_SECRET` — bei Rotation dieses Secrets werden alle Team-Sessions ungültig (gewollt).

### D-034 · Compliance-Linguistik in UI-Status-Texten (2026-06-07)

**Context:** Therapie-Performance-Claims wie „Präzision" oder „stark" triggern bei einem MDR-Audit das Medizinprodukt-Risiko (Klasse IIa). Das gilt auch für Score-Badges die implizit Therapie-Erfolg signalisieren.

**Decision:** Alle User-facing Status- und Score-Anzeigen verwenden rein deskriptive Sprache die Datenkonsistenz beschreibt, nicht Therapie-Erfolg. Verbotene Wörter: „Präzision" (für Bolus/Therapie), „stark" / „schwach" als Wertung, „gut" / „schlecht" als Therapie-Wertung. Erlaubt: „Konsistenz", „Verlauf", „stabil", „konsistent" (rein deskriptiv). ADAPT-Score Tier-Labels: „IM AUFBAU" / „STABIL" / „KONSISTENT" statt „IM AUFBAU" / „GUT" / „STARK". Footer-Nav: „LOG" statt „Einträge" (international, kein DE-only Begriff).

**Why:** Bei MDR-Einreichung gelten Status-Texte die Therapieerfolg suggerieren als Werbeaussagen über ein Medizinprodukt. Der Compliance Backlog (replit.md) nennt bereits ATTD-Quellen und Disclaimer-Anforderungen — dieser Grundsatz schließt die Lücke auf der Label-Ebene.

**Nicht wieder öffnen:** Wörter wie „Bolus-Präzision", „stark", „gut" sind dauerhaft aus User-facing Score-Anzeigen verbannt. Die Regel gilt für alle neuen Tiles, Badges und Status-Chips die Score/Performance zeigen — immer Datenkonsistenz-Framing wählen, nie Therapie-Erfolgs-Framing.

### D-033 · AI-Feedback-Funnel statt Email-Support (2026-06-07)

**Context:** Email-Support skaliert nicht. Glev AI bekommt sowieso täglich User-Inputs — Feedback-Intention in natürlicher Sprache erkennen ist trivial.

**Decision:** User-Feedback fließt durch die AI mit strukturierter Erfassung (what_noticed, where_noticed, what_broken, what_wished + category + severity) in `user_feedback`-Tabelle. Admin-Panel-Triage unter `/glev-ops/feedback` ersetzt Email-Triage für kleine bis mittlere Issues. `submit_structured_feedback`-Tool: direkter DB-Insert (kein Confirmation-Gate — Feedback ist keine medizinische Aktion, kein D-003-Scope). RLS: User kann INSERT, kein SELECT (Privacy). Admin bekommt Email via das Panel.

**Trade-offs:** AI muss richtig klassifizieren (bug vs. feature vs. complaint). Bei Mistral-Fehlern landet zu viel/wenig in der DB. Severity-Auto-Classification könnte Fehlerquelle sein — Admin korrigiert manuell via Status-Dropdown. Kein vollständiger Ersatz für komplexe Bug-Reports die Reproduce-Steps brauchen — die müssen weiter per Email kommen.

**Nicht wieder öffnen:** `submit_structured_feedback` braucht kein Confirmation-Gate. Feedback-Daten sind nicht medizinisch, keine D-003-Regel anwendbar. Compliance-Guard in Prompt: Therapie-Empfehlungs-Anfragen werden zu „kein Medizinprodukt"-Hinweis umgeleitet, NIE als Feedback erfasst.

### D-032 · Capacitor 8 Custom-Plugin-Registrierung via packageClassList (2026-06-07)

Inline-Plugins (Swift-Klassen im App-Target, nicht über ein separates SPM-Package eingebunden) werden von Capacitor 8 **nicht** auto-discovered. Sie müssen explizit in `capacitor.config.ts` (Repo-Root) und `ios/App/App/capacitor.config.json` (synced) unter dem root-level Array `packageClassList` gelistet sein. Fehlt der Eintrag, returnt jeder JS→Native-Call sofort `{"code":"UNIMPLEMENTED"}`, und die Capacitor-Bridge fällt silently auf die in `registerPlugin(name, { web: {...} })` definierte Web-Implementation zurück. Auf iOS äußert sich das als: kein Permission-Dialog, kein System-Toggle in Einstellungen → Mitteilungen → Glev, granted=false.

**Konkrete Falle (GlevCriticalAlertsPlugin, 2026-06-07):** `App.entitlements` hatte `com.apple.developer.usernotifications.critical-alerts=true`, Apple-Approval (Request X4854X3M8P) lag seit 2026-06-06 00:46 vor, das Provisioning-Profile vom 2026-06-06 01:21 sowie das signierte Binary enthielten das Entitlement (verifiziert mit `codesign -d --entitlements :- App.app`). Die Plugin-Klasse war `@objc(GlevCriticalAlertsPlugin) public class … : CAPPlugin, CAPBridgedPlugin` mit korrektem `jsName = "GlevCriticalAlerts"`. Trotzdem kein Dialog — weil `packageClassList` den Eintrag `"GlevCriticalAlertsPlugin"` nicht enthielt. Symptom im Xcode-Console-Log war eindeutig: `[criticalAlerts] native requestPermission failed: {"code":"UNIMPLEMENTED"}`.

**Fix:** In `capacitor.config.ts` UND `ios/App/App/capacitor.config.json` ALLE iOS-Native-Plugin-Klassennamen explizit listen — Standard-Capacitor-Plugins (HapticsPlugin, PushNotificationsPlugin, LocalNotificationsPlugin, ScreenOrientationPlugin, SharePlugin, HealthPlugin) UND Custom-Inline-Plugins (GlevCriticalAlertsPlugin). Reihenfolge irrelevant. Beide Files müssen konsistent sein, sonst überschreibt `npx cap sync ios` die JSON-Version mit dem (möglicherweise unvollständigen) TS-Stand.

**Nicht wieder öffnen:** Jedes neue native iOS-Plugin braucht einen `packageClassList`-Eintrag in BEIDEN Configs. Wenn ein Custom-Plugin auf iOS silent fehlschlägt (kein Dialog, kein Error, granted=false), erst `grep -c '<PluginName>' capacitor.config.ts ios/App/App/capacitor.config.json` prüfen — beide müssen 1 zurückgeben. Entitlements, Provisioning, Code-Signing und Swift-Class-Aufbau sind erst NACH dieser Prüfung verdächtig.

### D-029 · Foreground Alarm Ticker: DB-autoritativer Settings-Read pro Tick, Hypo immer first, kein Cross-Cooldown (2026-06-06)

**Anlass:** Safety-Critical Bug — Hyper-Alarm feuerte nicht trotz Wert 183 bei Threshold 180 (7+ Minuten App offen). Root Cause: fire-and-forget DB-Sync bei Mount + swallowed catch → localStorage blieb auf Default `{ enabled: false }` → stumme Miss.

**Architektur-Regel:**
1. **DB-autoritativer Read auf jeden Tick:** `checkLatestCgm()` liest alle drei Alarm-Settings via `Promise.allSettled([fetchLow, fetchElevated, fetchHigh])`. Bei DB-Fehler: graceful fallback auf localStorage (letzter bekannter Wert). DB-Ergebnis wird nach jedem erfolgreichen Read in localStorage gecacht (Offline-Fallback).
2. **Hypo-Check IMMER ZUERST:** `checkAndFireIfLow()` wird vor `checkAndFireIfElevated()` und `checkAndFireIfHyper()` aufgerufen. Reihenfolge ist hardcoded in `checkLatestCgm()`, keine Condition die Hypo überspringen oder verzögern kann.
3. **Kein Cross-Cooldown:** Jede Alarm-Stufe hat einen eigenen localStorage-Key (`glev_low_alarm_last_fired`, `glev_elevated_alarm_last_fired`, `glev_hyper_alarm_last_fired`). Ein Elevated-Alarm-Cooldown blockiert NIEMALS den Hyper-Alarm oder vice versa.
4. **Separate `enabled`-Checks:** Jede Alarm-Stufe wird nur gefeuert wenn ihre eigene `enabled`-Flag aus DB true ist. Kein hierarchischer Early-Return der nachfolgende Stufen überspringt.

**Nicht wieder öffnen:** Wenn neue Alarm-Typen hinzugefügt werden: immer mit eigenem Cooldown-Key + eigenem enabled-Flag + DB-Read auf jeden Tick. Nie über localStorage-only-State für safety-kritische Alarm-Enablement-Checks.

### D-028 · Chat Timeout + Mistral 429 Handling (2026-06-05)

`app/api/ai/chat/route.ts` hat zwei neue Fehler-Pfade: (1) **18 s Server-Timeout** via `AbortController` + `setTimeout` — überschreitet Mistral die Frist, schließt der Stream mit einem `{ error_code: "CHAT_TIMEOUT", retry_allowed: true }`-SSE-Frame; (2) **Mistral 429-Retry mit Back-off** via `callMistralWithRetry()` — bis zu 2 Retries (`MAX_MISTRAL_RETRIES=2`) mit maximal 8 s Wartezeit (`MAX_RETRY_WAIT_MS=8000`, Vercel-Function-Limit); nach Ausschöpfung sendet der Stream einen `{ error_code: "MISTRAL_RATE_LIMITED", retry_after_sec, retry_allowed: true }`-Frame. Beide Werte (`timeoutMs`, `sleep`) sind per `ChatDeps` injizierbar — ausschließlich für Unit-Tests. Client-seitig (`lib/useGlevAI.ts`) neu: `isSlow`-Flag (nach 15 s), `rateLimitCountdown`-State + `useEffect`-Ticker + Auto-Retry bei erstem 429-Treffer.

**Nicht wieder öffnen:** `MAX_RETRY_WAIT_MS=8000` ist hart an das Vercel Serverless Function Timeout (10 s) gekoppelt — niemals ohne Vercel-Limit-Prüfung erhöhen. `callMistralWithRetry` ist nur für 429 zuständig — alle anderen Status-Codes propagieren sofort weiter.
