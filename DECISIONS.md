### D-001 ¬Ζ Password-reset redirectTo must be /auth/confirm for BOTH routes (CORRECTED 2026-06-04)

> **βö†οΗè Rewritten 2026-06-04 with end-to-end evidence.** Earlier versions of D-001
> claimed the admin route needed `/auth/callback` (a "PKCE code exchange"). That is
> **wrong** βÄî `admin.generateLink({type:"recovery"})` always uses the implicit/hash
> flow; there is no `?code`. Routing recovery through `/auth/callback` is the bug,
> not the fix. Both routes must use `/auth/confirm`.

This project uses **Supabase Implicit Flow** (no PKCE toggle in the Email provider settings). `admin.generateLink({ type: "recovery" })` βÄî used by BOTH the admin panel and the self-service route βÄî produces a **hash-based** token. Supabase's `/verify` endpoint validates the OTP server-side and redirects to `redirectTo` with the session as a URL **hash fragment**. Hash fragments are browser-only βÄî a server route never sees them.

**Empirical proof (2026-06-04):** a throwaway user + `admin.generateLink({type:"recovery"})` + following the action_link, inspecting the redirect `Location`:
- `redirectTo = βÄΠ/auth/confirm` βÜ£ `βÄΠ/auth/confirm#access_token=βÄΠ&type=recovery` β€Ö
- `redirectTo = βÄΠ/auth/callback?next=/auth/confirm` βÜ£ `βÄΠ/auth/callback#access_token=βÄΠ` βÄî a **server** route that cannot read the `#hash`, finds no `?code`, and bounces the user to an error/root URL (the "lands on `glev.app#`" symptom).

**The required redirect URL βÄî BOTH routes:**
```
${appUrl}/auth/confirm
```
- `app/glev-ops/users/actions.ts` (admin recovery + invite)
- `app/api/auth/password-reset/route.ts` (self-service)

**Why /auth/confirm works:** it is a **client** page (`app/auth/confirm/page.tsx`) that manually parses the `#access_token` hash and calls `setSession()` (see the "Implicit/hash recovery flow" block). `detectSessionInUrl` is disabled on that page (`lib/supabase.ts`), so this manual hash handler is the **load-bearing piece** βÄî and it was the missing piece during all four reversals (Tasks #1152 βÜ£ #1171 βÜ£ #1179 βÜ£ #1187). Both candidate URLs were broken without it, so each reversal looked like the other URL was "the fix".

**Nicht wieder ΟΕffnen:** Do not route recovery/invite through `/auth/callback`. If reset breaks again, first re-run the E2E redirect check (generate a recovery link, follow it, inspect the `Location` header) before touching anything.

**See also:** `app/auth/confirm/page.tsx` (hash handler), `lib/supabase.ts` (detectSessionInUrl), `tests/unit/passwordResetRedirectTo.test.ts` and `tests/unit/passwordResetRoute.test.ts`.

**Enforcement-Architektur:**

**Kritischer Sound:** Eigene WAV (`glev_low_alarm.wav`), muss als `.caf` ins Bundle (Phase B). APNs verlangt bundled sound file fΟΦr critical alerts βÄî Supabase Storage reicht nicht.

**Nicht wieder ΟΕffnen:** Elevated darf nie auf `critical` hochgestuft werden, auch wenn User darum bittet βÄî Apple's Entitlement-Approval ist explizit auf Hypo/Severe-Hyper beschrΟΛnkt. Jede Ο³nderung des Scopes erfordert neue Apple-Submission.

### D-029 ¬Ζ Foreground Alarm Ticker: DB-autoritativer Settings-Read pro Tick, Hypo immer first, kein Cross-Cooldown (2026-06-06)

**Anlass:** Safety-Critical Bug βÄî Hyper-Alarm feuerte nicht trotz Wert 183 bei Threshold 180 (7+ Minuten App offen). Root Cause: fire-and-forget DB-Sync bei Mount + swallowed catch βÜ£ localStorage blieb auf Default `{ enabled: false }` βÜ£ stumme Miss.

**Architektur-Regel:**
1. **DB-autoritativer Read auf jeden Tick:** `checkLatestCgm()` liest alle drei Alarm-Settings via `Promise.allSettled([fetchLow, fetchElevated, fetchHigh])`. Bei DB-Fehler: graceful fallback auf localStorage (letzter bekannter Wert). DB-Ergebnis wird nach jedem erfolgreichen Read in localStorage gecacht (Offline-Fallback).
2. **Hypo-Check IMMER ZUERST:** `checkAndFireIfLow()` wird vor `checkAndFireIfElevated()` und `checkAndFireIfHyper()` aufgerufen. Reihenfolge ist hardcoded in `checkLatestCgm()`, keine Condition die Hypo ΟΦberspringen oder verzΟΕgern kann.
3. **Kein Cross-Cooldown:** Jede Alarm-Stufe hat einen eigenen localStorage-Key (`glev_low_alarm_last_fired`, `glev_elevated_alarm_last_fired`, `glev_hyper_alarm_last_fired`). Ein Elevated-Alarm-Cooldown blockiert NIEMALS den Hyper-Alarm oder vice versa.
