# Decisions and Changes (DECISIONS.md)

## Decisions

### D-001 ¬∑ Password-reset redirectTo must route through /auth/callback (2026-06-04)

`admin.generateLink({ type: "recovery" })` produces a **hash-based** session token. When Supabase processes the link and redirects to the `redirectTo` URL, it appends the token as a URL *hash fragment* (`#access_token=‚Ä¶`). Hash fragments are client-side only ‚Äî the Supabase JS SDK immediately consumes and strips them, leaving the user on `/#` instead of the password-reset form.

**The required redirect URL for the admin panel (`app/glev-ops/users/actions.ts`):**
```
${appUrl}/auth/callback?next=/auth/confirm
```

**Why the hop through `/auth/callback`:**
Our server-side handler (`app/auth/callback/route.ts`) calls `supabase.auth.exchangeCodeForSession()`, which resolves the hash token server-side and sets a proper session cookie. It then redirects to `/auth/confirm?session=ready&type=recovery` ‚Äî a clean URL the page can render from.

**Contrast with `resetPasswordForEmail()` (PKCE flow):**
That method sends a PKCE code as a query-string parameter (`?code=‚Ä¶`), not a hash. Query params survive server-side routing, so `/auth/confirm` can handle them directly without the callback hop.

**History:** This URL was toggled four times (Tasks #1152 ‚Üí #1171 ‚Üí #1179 ‚Üí #1187), breaking real users' reset emails each time. The self-service route (`app/api/auth/password-reset/route.ts`) currently uses `/auth/confirm` directly and should be watched ‚Äî if it breaks, switch it to `/auth/callback?next=/auth/confirm` as well.

**Nicht wieder √∂ffnen:** Do not simplify the admin-panel `redirectTo` back to `/auth/confirm` without first verifying that real users can complete the reset flow end-to-end in production. The `/auth/callback` hop is load-bearing.

**See also:** `tests/unit/passwordResetRedirectTo.test.ts` ‚Äî schema-contract tests that catch accidental URL reversions.
