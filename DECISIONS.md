## Decisions
> Was hier steht, ist eine bewusste Nicht-Entscheidung oder eine Weichenstellung, die nicht mehr offen diskutiert werden soll.
### D-001 ¬Ζ Supabase statt eigener Auth/DB-Infrastruktur (2026-01-15)
Supabase liefert PostgreSQL, Row Level Security und Auth (Email/Password + JWT) als verwalteten Service. Eine eigene Postgres-Instanz mit Auth-Stack hΟΛtte dieselbe Betriebslast ohne den Mehrwert. Firebase wurde geprΟΦft, aber wegen SQL-Querybarkeit und Open-Source-KompatibilitΟΛt verworfen. **Nicht wieder ΟΕffnen:** Solange keine Multi-Tenant-SaaS-Isolation erforderlich ist, bleibt Supabase die Datenebene.
### D-002 ¬Ζ Capacitor statt React Native fΟΦr mobile Shells (2026-01-20)
Glev ist primΟΛr eine Next.js-Web-App. Capacitor erlaubt es, exakt dieselbe Web-Codebasis als Thin-Webview-Shell auf iOS/Android zu verpacken βÄî kein zweiter Rendering-Pfad, keine doppelte Feature-Implementierung. React Native hΟΛtte eine separate Komponentenbibliothek und Logik-Schicht erfordert. **Nicht wieder ΟΕffnen:** Solange die App keine tief nativen UI-Patterns (z. B. UIKit-Navigation) braucht, bleibt Capacitor die Native-Schicht.
### D-003 ¬Ζ Keine direkten Dosis-Anweisungen in der UI (2026-02-01)
Glev ist ein Entscheidungs-*Support*-System, kein Medizinprodukt Klasse IIb. Direkte Dosis-Anweisungen (z. B. βÄûNimm jetzt 4 IE") wΟΦrden eine klinische Validierung und MDR-Zertifizierung erfordern, die den aktuellen Rahmen sprengen. Alle Engine-Empfehlungen sind als GesprΟΛchsbasis fΟΦrs Diabetes-Team gerahmt. **Nicht wieder ΟΕffnen:** Selbst nach einer mΟΕglichen MDR-Einreichung braucht jede Ο³nderung an diesem Prinzip eine explizite Freigabe durch das medizinische Verantwortungsteam.
### D-004 ¬Ζ next-intl statt i18next fΟΦr Lokalisierung (2026-02-10)
`next-intl` ist fΟΦr Next.js App Router (Server Components, `generateMetadata`, Middleware-basiertes Locale-Routing) nativ ausgelegt. i18next benΟΕtigt zusΟΛtzliche Adapter (`next-i18next`) und arbeitet primΟΛr Client-seitig, was SSR-Hydration-Mismatches erzeugt. ### D-005 ¬Ζ Pump-Dosierung bewusst aus dem Engine-Scope ausgeschlossen (2026-02-15)
Die Glev Engine berechnet Bolusempfehlungen fΟΦr ICT-Nutzer (Pen-Injektionen). Insulinpumpen-Nutzer haben eigene Basal-/Bolusprofile, die von der Engine nicht modelliert werden. ### D-007 ¬Ζ Engine-Algorithmus: gewichteter k-NN-Ansatz statt LLM-only (2026-05-20)
**Warum kein LLM-only-Approach?**
Ein LLM kΟΕnnte prinzipiell aus dem Freitext-Meal-Log direkt eine Dosisempfehlung generieren. Das wurde bewusst verworfen:
- **Nicht deterministisch:** Gleiche Eingabe, unterschiedliche Ausgabe βÄî bei klinischen Empfehlungen nicht akzeptabel.
- **Nicht personalisierbar:** Ein generisches LLM kennt nicht das individuelle ICR, die InsulinsensitivitΟΛt oder das Tageszeit-Muster des Nutzers.
- **Nicht auditierbar:** Der Reasoning-Pfad eines LLM ist fΟΦr Compliance-Zwecke (MDR) nicht hinreichend dokumentierbar.
- **Latenz & Kosten:** Jede Dosisempfehlung wΟΦrde einen API-Call erfordern; beim gewichteten Ansatz ist das eine lokale Pure-Funktion.
**GewΟΛhlter Ansatz: personalisierter, gewichteter Durchschnitt (k-NN-ΟΛhnlich)**
Die Engine lernt den ICR des Nutzers aus finalisierten Mahlzeiten (`carbs / insulin`) und gewichtet schlechtere Outcomes (OVERDOSE, UNDERDOSE βÜ£ 0.3; SPIKE βÜ£ 0.7) geringer als GOOD (1.0). Das ist konzeptionell verwandt mit einem gewichteten k-Nearest-Neighbour, aber ohne explizite Feature-Distanz βÄî der Nutzungskontext (Tageszeit, Meal-Type) wird ΟΦber separate Buckets abgedeckt, nicht ΟΦber eine Ο³hnlichkeitsmetrik.
**Was das LLM tut (und was nicht):**
- GPT (`lib/ai/systemPrompt.ts`) parst den Freitext-Meal-Log und klassifiziert die Mahlzeit in einen der vier Typen (FAST_CARBS / HIGH_FAT / HIGH_PROTEIN / BALANCED).
- GPT schlΟΛgt **niemals** eine Dosis vor und wertet **niemals** eine Dosis aus βÄî das ist im System-Prompt explizit verboten.
- Die Klassifikation flieΟüt in die Spike-Cutoffs (FAST_CARBS=70, HIGH_FAT=40, HIGH_PROTEIN=50, BALANCED=55 mg/dL) und die Meal-Type-Farb-/Label-Darstellung.
**VollstΟΛndige Algorithmus-Dokumentation:** `docs/engine-algorithm.md`
**Nicht wieder ΟΕffnen:** Solange Glev kein Klasse-IIb-Medizinprodukt ist und keine klinische Validierung eines LLM-Dosierungsmodells vorliegt, bleibt die deterministische, auditierbare Pure-Function-Pipeline die einzige Dosis-Logik.
### D-009 ¬Ζ Telegram als Agent-Kommunikationskanal (Message-Bus via Supabase) (2026-05-20)
Der Agent (Replit) kann Lucas Fragen stellen und auf Antworten warten, ohne dass Lucas Replit ΟΕffnen muss. Der Kanal besteht aus drei Teilen:
1. **Supabase-Tabelle `agent_messages`** βÄî `outbound`-Zeilen werden vom Script geschrieben, `inbound`-Zeilen vom Webhook-Receiver. RLS: service role only (kein anon/authenticated-Zugriff).
2. **`scripts/notify-telegram.mjs`** βÄî ESM-Script, schreibt `outbound`-Zeile, schickt Telegram-Nachricht, wartet via Supabase-Realtime-Subscription auf `inbound`-Antwort (max. 10 Minuten, dann `TIMEOUT`).
3. **`app/api/telegram/webhook/route.ts`** βÄî Next.js-Route, prΟΦft `X-Telegram-Bot-Api-Secret-Token`, transkribiert Voice-Nachrichten via OpenAI Whisper (`whisper-1`), extrahiert `task_id` per Regex aus dem Reply-Kontext, schreibt `inbound`-Zeile.
**Warum Telegram statt Email/SMS?** Niedrige Latenz (Sekunden statt Minuten), native Reply-Thread-UnterstΟΦtzung (kein neues Compose-Fenster), kein SMS-Provider-Setup. **Warum Supabase als Bus statt direktem Polling der Telegram-API?** Das Script hat keinen ΟΕffentlich erreichbaren Port und kann nicht als Webhook-Receiver fungieren. Supabase-Realtime ist the einzige zuverlΟΛssige Kanal zwischen dem Webhook (Vercel) und dem wartenden Script (Replit).
**Nicht wieder ΟΕffnen:** Solange der Agent auf Replit und die Produktion auf Vercel lΟΛuft, ist dieser hybride Ansatz (Webhook auf Vercel empfΟΛngt, Supabase-Bus transportiert, Script auf Replit wartet) die richtige Architektur. Eine direkte Polling-LΟΕsung wΟΦrde den Telegram-Timeout ΟΦberschreiten und eine Long-Poll-Infrastruktur erfordern.

### D-012 ¬Ζ Automatisierte DB-Migrations via GitHub Actions (2026-05-23)
`.github/workflows/apply-migration.yml` fΟΦhrt `scripts/apply-migration.mjs --all` nach jedem Push auf `main` aus (nur wenn Migrations- oder Script-Dateien geΟΛndert wurden). Die AusfΟΦhrung ist idempotent: eine neue `schema_migrations`-Tabelle in Supabase verfolgt, welche `.sql`-Dateien bereits angewendet wurden. Neue Dateien werden in alphabetischer (timestamp-basierter) Reihenfolge angewendet; bereits eingetragene Dateien werden ΟΦbersprungen.
**BenΟΕtigte GitHub-Repo-Secrets (einmalig einrichten):**
- `SUPABASE_ACCESS_TOKEN` βÄî Supabase Personal Access Token (https://supabase.com/dashboard/account/tokens)
- `NEXT_PUBLIC_SUPABASE_URL` βÄî Supabase-Projekt-URL (gleicher Wert wie in Vercel)
**Trigger-Strategie:** Der Workflow feuert auf jeden Push auf `main` βÄî ohne Pfad-Filter. Push auf `main` ist der Proxy for βÄûVercel deployed". Die Migration lΟΛuft damit nach jedem Deploy; wenn keine neuen Dateien vorliegen, beendet das Script in Sekunden mit βÄûnothing to do" (idempotent). Ein Pfad-Filter wΟΦrde das Ziel verfehlen: Migrationen mΟΦssen auch dann greifen, wenn ein reiner Code-Push (ohne neue `.sql`-Datei) einen Vercel-Deploy auslΟΕst.
**Bootstrap for bestehende Produktions-DBs (einmalig):** Beim ersten Aktivieren auf einer bestehenden Datenbank (mit bereits manuell angewendeten Migrationen) muss `--baseline` einmalig ausgefΟΦhrt werden βÄî entweder lokal (`pnpm db:migrate:baseline`) oder per GitHub Actions `workflow_dispatch` with `mode = baseline`. Das erstellt die Tracking-Tabelle und markiert alle vorhandenen `.sql`-Dateien als angewendet, OHNE SQL auszufΟΦhren. Danach erkennt `--all` nur neue Dateien als pending.
**Warum GitHub Actions statt Vercel post-deploy-Hook?** Vercel bietet keinen nativen post-deploy-Hook fΟΦr externe SQL-AusfΟΦhrung. GitHub Actions hat bereits Zugriff auf Repo-Secrets, lΟΛuft nach jedem Push auf `main` (zeitgleich mit dem Vercel-Deploy-Trigger) und ist in der bestehenden CI-Infrastruktur integriert.
**Nicht wieder ΟΕffnen:** Die `schema_migrations`-Tabelle ist die einzige Migration-Tracking-Quelle. Niemals manuell Zeilen lΟΕschen oder die Tabelle droppen βÄî das wΟΦrde bereits angewendete Migrationen erneut ausfΟΦhren. Wenn eine Migration rΟΦckgΟΛngig gemacht werden soll, eine neue `*_rollback.sql`-Datei erstellen.

**Explizite Nicht-Entscheidungen:**

### D-014 ¬Ζ Glev-AI-Rate-Limit in Supabase statt In-Memory-Map (2026-05-23)
Die 20-Requests-pro-Minute-Grenze in `POST /api/ai/chat` lebt jetzt in der neuen Tabelle `public.ai_rate_limit_hits` (`user_id UUID`, `hit_at TIMESTAMPTZ DEFAULT NOW()`, Index auf `(user_id, hit_at DESC)`, RLS an, keine Policies βÄî nur Service-Role-Zugriff). Migration: `supabase/migrations/20260523_ai_rate_limit_hits.sql`. **Warum Supabase und nicht Upstash/Redis?** Wir haben Supabase bereits in der Hand (D-001), Service-Role-Client (`lib/supabaseAdmin.ts`) ist ΟΦberall verkabelt, und 20/min pro User ist keine Hochfrequenz-Last, die einen dedizierten Key-Value-Store rechtfertigt. Upstash wΟΦrde ein zusΟΛtzliches Secret, eine zusΟΛtzliche Rechnungsposition und eine zusΟΛtzliche AusfalldomΟΛne bedeuten.
**Fail-open-Politik:** Wenn die Admin-Client-Erzeugung wirft (kein Service-Key) oder das Count/Insert-Query einen Fehler liefert, gibt `isRateLimited` `false` zurΟΦck. BegrΟΦndung: ein DB-Hiccup darf die UI nicht hart zumachen; das Mistral-seitige Konto-Limit bleibt als zweite Schutzschicht.
**Cleanup:** Jeder erfolgreiche Insert feuert opportunistisch ein `DELETE WHERE hit_at < cutoff` fΟΦr denselben User (best-effort, kein await auf Result). Damit bleibt die Tabelle proportional zur aktiven Userzahl, ohne dass ein separater Cron nΟΕtig wΟΛre.
**Nicht wieder ΟΕffnen:** Die Werte `RATE_LIMIT_MAX = 20` und `RATE_LIMIT_WINDOW_MS = 60_000` spiegeln das in der 429-Response sichtbare Versprechen ("max 20 requests per minute") wider βÄî Ο³nderungen mΟΦssen auch die Error-Copy anpassen. Migration zu Upstash erst, wenn die Tabelle messbar warm wird (βâΪ1 Insert/s/Instance) oder Mistral-Quoten-Konflikte mit Multi-Tenant-Bursts auftreten.

### D-016 ¬Ζ Glev AI Phase 3 βÄî Function-Calling-Architektur (Tools server-side, READ-only in Task 1) (2026-05-24)

### D-025 ¬Ζ Facebook Conversion API (CAPI): Architektur-Entscheidungen (2026-05-27)

**Kontext:** Werbe-Tracking fΟΦr Meta-Kampagnen. DSGVO + iOS14+ schrΟΛnken Browser-seitiges Pixel-Tracking stark ein βÄî Server-seitiges CAPI ist die einzige zuverlΟΛssige Alternative.

**Drei Architektur-Entscheidungen:**

1. **Browser-Pixel + Server-CAPI parallel (Deduplication via `event_id`):**

Das Banner zeigt sich beim Erstbesuch (600ms Delay fΟΦr SSR-Flash-Prevention), danach wird der gespeicherte Consent still wiederhergestellt.

3. **Webhook-CAPI ist fire-and-forget βÄî blockiert nie den Stripe-Return:**
   `sendCapiEvent()` in Stripe-Webhooks ist `.catch()`-gewrappt. Meta-API-Fehler sind kein Grund fΟΦr einen Stripe-Webhook-Retry (502/503 wΟΦrde Stripe bis zu 3 Tage lang wiederholen, mit jeder Retry ein doppelter DB-Write). **Plan-Differenzierung in Stripe-Metadata:**

**Nicht wieder ΟΕffnen:** Alle CAPI-Calls laufen ΟΦber `lib/fb-capi-server.ts` (Server) bzw. `lib/fb-capi-client.ts` + `/api/fb-capi` (Browser-Proxy). Keine direkten Graph-API-Calls auΟüerhalb dieser Dateien. Consent-Check nicht umgehen βÄî auch nicht fΟΦr A/B-Tests oder Drittanbieter.
