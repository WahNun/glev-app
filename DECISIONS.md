### D-001 ¬∑ Password-reset redirectTo must be /auth/confirm for BOTH routes (CORRECTED 2026-06-04)

> **‚öÝÔ∏è Rewritten 2026-06-04 with end-to-end evidence.** Earlier versions of D-001
> claimed the admin route needed `/auth/callback` (a "PKCE code exchange"). That is
> **wrong** ‚Äî `admin.generateLink({type:"recovery"})` always uses the implicit/hash
> flow; there is no `?code`. Routing recovery through `/auth/callback` is the bug,
> not the fix. Both routes must use `/auth/confirm`.

This project uses **Supabase Implicit Flow** (no PKCE toggle in the Email provider settings). `admin.generateLink({ type: "recovery" })` ‚Äî used by BOTH the admin panel and the self-service route ‚Äî produces a **hash-based** token. Supabase's `/verify` endpoint validates the OTP server-side and redirects to `redirectTo` with the session as a URL **hash fragment**. Hash fragments are browser-only ‚Äî a server route never sees them.

**Empirical proof (2026-06-04):** a throwaway user + `admin.generateLink({type:"recovery"})` + following the action_link, inspecting the redirect `Location`:
- `redirectTo = ‚Ä¶/auth/confirm` ‚Üí `‚Ä¶/auth/confirm#access_token=‚Ä¶&type=recovery` ‚úÖ
- `redirectTo = ‚Ä¶/auth/callback?next=/auth/confirm` ‚Üí `‚Ä¶/auth/callback#access_token=‚Ä¶` ‚Äî a **server** route that cannot read the `#hash`, finds no `?code`, and bounces the user to an error/root URL (the "lands on `glev.app#`" symptom).

**The required redirect URL ‚Äî BOTH routes:**
```
${appUrl}/auth/confirm
```
- `app/glev-ops/users/actions.ts` (admin recovery + invite)
- `app/api/auth/password-reset/route.ts` (self-service)

**Why /auth/confirm works:** it is a **client** page (`app/auth/confirm/page.tsx`) that manually parses the `#access_token` hash and calls `setSession()` (see the "Implicit/hash recovery flow" block). `detectSessionInUrl` is disabled on that page (`lib/supabase.ts`), so this manual hash handler is the **load-bearing piece** ‚Äî and it was the missing piece during all four reversals (Tasks #1152 ‚Üí #1171 ‚Üí #1179 ‚Üí #1187). Both candidate URLs were broken without it, so each reversal looked like the other URL was "the fix".

**Nicht wieder √∂ffnen:** Do not route recovery/invite through `/auth/callback`. If reset breaks again, first re-run the E2E redirect check (generate a recovery link, follow it, inspect the `Location` header) before touching anything.

**See also:** `app/auth/confirm/page.tsx` (hash handler), `lib/supabase.ts` (detectSessionInUrl), `tests/unit/passwordResetRedirectTo.test.ts` and `tests/unit/passwordResetRoute.test.ts`.
