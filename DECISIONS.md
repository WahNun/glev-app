### D-001 В· Supabase statt eigener Auth/DB-Infrastruktur (2026-01-15)
Supabase liefert PostgreSQL, Row Level Security und Auth (Email/Password + JWT) als verwalteten Service. Eine eigene Postgres-Instanz mit Auth-Stack hГӨtte dieselbe Betriebslast ohne den Mehrwert. ### D-002 В· Capacitor statt React Native fГјr mobile Shells (2026-01-20)
Glev ist primГӨr eine Next.js-Web-App. Capacitor erlaubt es, exakt dieselbe Web-Codebasis als Thin-Webview-Shell auf iOS/Android zu verpacken вҖ” kein zweiter Rendering-Pfad, keine doppelte Feature-Implementierung. ### D-003 В· Keine direkten Dosis-Anweisungen in der UI (2026-02-01)

### D-007 В· Engine-Algorithmus: gewichteter k-NN-Ansatz statt LLM-only (2026-05-20)
**Warum kein LLM-only-Approach?**
Ein LLM kГ¶nnte prinzipiell aus dem Freitext-Meal-Log direkt eine Dosisempfehlung generieren. Das wurde bewusst verworfen:
- **Nicht deterministisch:** Gleiche Eingabe, unterschiedliche Ausgabe вҖ” bei klinischen Empfehlungen nicht akzeptabel.
- **Nicht personalisierbar:** Ein generisches LLM kennt nicht das individuelle ICR, die InsulinsensitivitГӨt oder das Tageszeit-Muster des Nutzers.
- **Nicht auditierbar:** Der Reasoning-Pfad eines LLM ist fГјr Compliance-Zwecke (MDR) nicht hinreichend dokumentierbar.
- **Latenz & Kosten:** Jede Dosisempfehlung wГјrde einen API-Call erfordern; beim gewichteten Ansatz ist das eine lokale Pure-Funktion.
**GewГӨhlter Ansatz: personalisierter, gewichteter Durchschnitt (k-NN-ГӨhnlich)**
Die Engine lernt den ICR des Nutzers aus finalisierten Mahlzeiten (`carbs / insulin`) und gewichtet schlechtere Outcomes (OVERDOSE, UNDERDOSE вҶ’ 0.3; SPIKE вҶ’ 0.7) geringer als GOOD (1.0). Das ist konzeptionell verwandt mit einem gewichteten k-Nearest-Neighbour, aber ohne explizite Feature-Distanz вҖ” der Nutzungskontext (Tageszeit, Meal-Type) wird Гјber separate Buckets abgedeckt, nicht Гјber eine Г„hnlichkeitsmetrik.
**Was das LLM tut (und was nicht):**
- GPT (`lib/ai/systemPrompt.ts`) parst den Freitext-Meal-Log und klassifiziert die Mahlzeit in einen der vier Typen (FAST_CARBS / HIGH_FAT / HIGH_PROTEIN / BALANCED).
- GPT schlГӨgt **niemals** eine Dosis vor und wertet **niemals** eine Dosis aus вҖ” das ist im System-Prompt explizit verboten.
- Die Klassifikation flieГҹt in die Spike-Cutoffs (FAST_CARBS=70, HIGH_FAT=40, HIGH_PROTEIN=50, BALANCED=55 mg/dL) und die Meal-Type-Farb-/Label-Darstellung.
**VollstГӨndige Algorithmus-Dokumentation:** `docs/engine-algorithm.md`
**Nicht wieder Г¶ffnen:** Solange Glev kein Klasse-IIb-Medizinprodukt ist und keine klinische Validierung eines LLM-Dosierungsmodells vorliegt, bleibt die deterministische, auditierbare Pure-Function-Pipeline die einzige Dosis-Logik.
### D-009 В· Telegram als Agent-Kommunikationskanal (Message-Bus via Supabase) (2026-05-20)
Der Agent (Replit) kann Lucas Fragen stellen und auf Antworten warten, ohne dass Lucas Replit Г¶ffnen muss. Der Kanal besteht aus drei Teilen:
1. **Supabase-Tabelle `agent_messages`** вҖ” `outbound`-Zeilen werden vom Script geschrieben, `inbound`-Zeilen vom Webhook-Receiver. RLS: service role only (kein anon/authenticated-Zugriff).

**Warum Telegram statt Email/SMS?** Niedrige Latenz (Sekunden statt Minuten), native Reply-Thread-UnterstГјtzung (kein neues Compose-Fenster), kein SMS-Provider-Setup. **Warum Supabase als Bus statt direktem Polling der Telegram-API?** Das Script hat keinen Г¶ffentlich erreichbaren Port und kann nicht als Webhook-Receiver fungieren. Supabase-Realtime ist the einzige zuverlГӨssige Kanal zwischen dem Webhook (Vercel) und dem wartenden Script (Replit).
**Nicht wieder Г¶ffnen:** Solange der Agent auf Replit und die Produktion auf Vercel lГӨuft, ist dieser hybride Ansatz (Webhook auf Vercel empfГӨngt, Supabase-Bus transportiert, Script auf Replit wartet) die richtige Architektur. ### D-012 В· Automatisierte DB-Migrations via GitHub Actions (2026-05-23)
`.github/workflows/apply-migration.yml` fГјhrt `scripts/apply-migration.mjs --all` nach jedem Push auf `main` aus (nur wenn Migrations- oder Script-Dateien geГӨndert wurden). **BenГ¶tigte GitHub-Repo-Secrets (einmalig einrichten):**
- `SUPABASE_ACCESS_TOKEN` вҖ” Supabase Personal Access Token (https://supabase.com/dashboard/account/tokens)
- `NEXT_PUBLIC_SUPABASE_URL` вҖ” Supabase-Projekt-URL (gleicher Wert wie in Vercel)
**Trigger-Strategie:** Der Workflow feuert auf jeden Push auf `main` вҖ” ohne Pfad-Filter. Push auf `main` ist der Proxy for вҖһVercel deployed". Die Migration lГӨuft damit nach jedem Deploy; wenn keine neuen Dateien vorliegen, beendet das Script in Sekunden mit вҖһnothing to do" (idempotent). Ein Pfad-Filter wГјrde das Ziel verfehlen: Migrationen mГјssen auch dann greifen, wenn ein reiner Code-Push (ohne neue `.sql`-Datei) einen Vercel-Deploy auslГ¶st.
**Bootstrap for bestehende Produktions-DBs (einmalig):** Beim ersten Aktivieren auf einer bestehenden Datenbank (mit bereits manuell angewendeten Migrationen) muss `--baseline` einmalig ausgefГјhrt werden вҖ” entweder lokal (`pnpm db:migrate:baseline`) oder per GitHub Actions `workflow_dispatch` with `mode = baseline`. Das erstellt die Tracking-Tabelle und markiert alle vorhandenen `.sql`-Dateien als angewendet, OHNE SQL auszufГјhren. Danach erkennt `--all` nur neue Dateien als pending.
**Warum GitHub Actions statt Vercel post-deploy-Hook?** Vercel bietet keinen nativen post-deploy-Hook fГјr externe SQL-AusfГјhrung. GitHub Actions hat bereits Zugriff auf Repo-Secrets, lГӨuft nach jedem Push auf `main` (zeitgleich mit dem Vercel-Deploy-Trigger) und ist in der bestehenden CI-Infrastruktur integriert.
**Nicht wieder Г¶ffnen:** Die `schema_migrations`-Tabelle ist die einzige Migration-Tracking-Quelle. Niemals manuell Zeilen lГ¶schen oder die Tabelle droppen вҖ” das wГјrde bereits angewendete Migrationen erneut ausfГјhren. Wenn eine Migration rГјckgГӨngig gemacht werden soll, eine neue `*_rollback.sql`-Datei erstellen.
### D-013 В· Glev AI Phase 2 вҖ” Consent auf profiles, Mistral als Chat-Provider (2026-05-23)
**Consent-Quelle:** Die UI liest und schreibt ab sofort `profiles.ai_consent_at TIMESTAMPTZ` + `profiles.ai_consent_version TEXT` (beide bereits in Supabase vorhanden). **Chat-Provider:** Mistral (`mistral-small-latest`, `maxTokens: 300`, `temperature: 0.4`) вҖ” bewusst separat vom OpenAI-Client (`lib/ai/openaiClient.ts`), der weiterhin die Food-Parser-Pipeline befeuert. **Explizite Nicht-Entscheidungen:**
- **Keine Supabase-Persistenz von Chat-VerlГӨufen.** Konversation lebt nur in `sessionStorage` (max. 10 Nachrichten), wird beim Session-Ende automatisch verworfen. BegrГјndung: niedrigster Datenschutz-Footprint solange Glev kein MDR-Produkt ist; Logs mГјssten sonst als вҖһmedizinische Aufzeichnung" behandelt werden. - **Kein Phase-2-Settings-Toggle zum Widerrufen des Consents.** Wer den Consent zurГјckziehen will, kann das aktuell nur Гјber einen Support-Request. Ein eigener Toggle wГјrde einen klaren Revoke-Flow + LГ¶schung der `sessionStorage`-History + UI-Strings + Test-Coverage in Settings erfordern вҖ” separater Task.
- **Dummy `contextSnapshot`** (`glucoseSummary`, `iobSummary`, `lastMealDescription`) wird vom `useGlevAI`-Hook gesendet, bis ein Follow-up die echten CGM/IOB/Letzte-Mahlzeit-Werte einbindet. Die API validiert nur die Shape, nicht die Inhalte.
**Sicherheit/Compliance:** `POST /api/ai/chat` ist hart durch vier Gates abgesichert: (1) 401 ohne authedClient-User, (2) 403 ohne `profiles.ai_consent_at`, (3) 429 bei >20 Requests/Minute/User (in-memory Map, per-Function-Instance), (4) 503 ohne `MISTRAL_API_KEY`. **Nicht wieder Г¶ffnen:** Der Consent-Mechanismus ist bewusst timestamped + versioned (nicht boolean), damit eine spГӨtere MDR-relevante Г„nderung des Disclaimer-Texts den User automatisch zu einem neuen Consent zwingen kann (`ai_consent_version` Vergleich gegen aktuelle Konstante). Provider-Wechsel weg von Mistral: nur `lib/ai/mistralClient.ts` + Stream-Loop in `app/api/ai/chat/route.ts` betroffen вҖ” Prompt, Consent und Hook bleiben unverГӨndert.
### D-014 В· Glev-AI-Rate-Limit in Supabase statt In-Memory-Map (2026-05-23)
Die 20-Requests-pro-Minute-Grenze in `POST /api/ai/chat` lebt jetzt in der neuen Tabelle `public.ai_rate_limit_hits` (`user_id UUID`, `hit_at TIMESTAMPTZ DEFAULT NOW()`, Index auf `(user_id, hit_at DESC)`, RLS an, keine Policies вҖ” nur Service-Role-Zugriff). **Warum Supabase und nicht Upstash/Redis?** Wir haben Supabase bereits in der Hand (D-001), Service-Role-Client (`lib/supabaseAdmin.ts`) ist Гјberall verkabelt, und 20/min pro User ist keine Hochfrequenz-Last, die einen dedizierten Key-Value-Store rechtfertigt. Upstash wГјrde ein zusГӨtzliches Secret, eine zusГӨtzliche Rechnungsposition und eine zusГӨtzliche AusfalldomГӨne bedeuten.
**Fail-open-Politik:** Wenn die Admin-Client-Erzeugung wirft (kein Service-Key) oder das Count/Insert-Query einen Fehler liefert, gibt `isRateLimited` `false` zurГјck. BegrГјndung: ein DB-Hiccup darf die UI nicht hart zumachen; das Mistral-seitige Konto-Limit bleibt als zweite Schutzschicht.
**Cleanup:** Jeder erfolgreiche Insert feuert opportunistisch ein `DELETE WHERE hit_at < cutoff` fГјr denselben User (best-effort, kein await auf Result). Damit bleibt die Tabelle proportional zur aktiven Userzahl, ohne dass ein separater Cron nГ¶tig wГӨre.
**Nicht wieder Г¶ffnen:** Die Werte `RATE_LIMIT_MAX = 20` und `RATE_LIMIT_WINDOW_MS = 60_000` spiegeln das in der 429-Response sichtbare Versprechen ("max 20 requests per minute") wider вҖ” Г„nderungen mГјssen auch die Error-Copy anpassen. Migration zu Upstash erst, wenn die Tabelle messbar warm wird (вү«1 Insert/s/Instance) oder Mistral-Quoten-Konflikte mit Multi-Tenant-Bursts auftreten.
