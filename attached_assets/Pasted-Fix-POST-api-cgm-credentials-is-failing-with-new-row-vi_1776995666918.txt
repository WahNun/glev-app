Fix: POST /api/cgm/credentials is failing with "new row violates row-level security policy". Root cause: DB write is using the user-scoped ssrClient instead of the service-role admin client.

STEP 1 — Open app/api/cgm/credentials/route.ts. The flow should be:
1. Call authenticate(req) → gets user (verifies identity, via cookie or bearer)
2. Use a SEPARATE service-role client for the actual upsert into cgm_credentials

Check lib/cgm/supabase.ts — it should export something like createAdminClient() or getAdminClient() that uses SUPABASE_SERVICE_ROLE_KEY. If it does, use that. If it doesn't, create the admin client inline:

import { createClient } from '@supabase/supabase-js';
const admin = createClient(
  process.env.SUPABASE_URL || process.env.NEXT_PUBLIC_SUPABASE_URL || '',
  process.env.SUPABASE_SERVICE_ROLE_KEY || ''
);

Then for the upsert:
const { error: upsertError } = await admin
  .from('cgm_credentials')
  .upsert({
    user_id: user.id,
    llu_email: email,
    llu_password_encrypted: encryptedPassword,
    llu_region: region || 'eu',
    updated_at: new Date().toISOString(),
  }, { onConflict: 'user_id' });

if (upsertError) throw new Error('supabase: ' + upsertError.message);

STEP 2 — Do the same audit on:
- app/api/cgm/credentials/route.ts DELETE handler (use admin client for .delete())
- app/api/cgm/latest/route.ts (reads cgm_credentials — also must use admin client OR the user's own session for SELECT)
- app/api/cgm/history/route.ts (same)

Rule of thumb:
- Authentication (verifying who the user is) = anon/ssr client with session
- DB writes to cgm_credentials = admin client (service role)
- DB reads from cgm_credentials = admin client is fine (since we already verified user.id from authenticate())

STEP 3 — npm run build
