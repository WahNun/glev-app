# Diagnose + Fix: Glev+ Webhook → Supabase

## Problem
Wenn ein Glev+-User bezahlt, zeigt /pro/success die Confirmation — aber es ist unklar ob
der Webhook die Supabase-Tabelle korrekt mit "plus_subscription" updatet.
Wenn nicht: User sieht Bestätigung aber hat keinen aktiven Plan.

## STEP 0 — Ist-Zustand verstehen

```bash
# Wo ist der Webhook?
find app/ -name "*.ts" | xargs grep -l "webhook\|stripe.*event\|checkout.session.completed" 2>/dev/null | grep -v node_modules

# Was macht er mit dem subscription type?
grep -n "plus_subscription\|subscription_type\|plan\|tier\|supabase.*update\|supabase.*insert" \
  $(find app/ -name "*.ts" | xargs grep -l "webhook" 2>/dev/null | grep -v node_modules) | head -40
```

Lies die Webhook-Datei(en) komplett durch.

## STEP 1 — Prüfe ob plus_subscription korrekt erkannt wird

Im Webhook-Handler: wenn `checkout.session.completed` eintrifft, muss der Code erkennen
ob es sich um Beta (S), Pro (M) oder Plus (L) handelt.

Typisches Pattern ist ein Vergleich der Price ID mit den env vars. Prüfe ob:

```ts
const priceId = session.line_items?.data[0]?.price?.id
  ?? session.metadata?.price_id;

const plan =
  priceId === process.env.STRIPE_BETA_PRICE_ID  ? 'beta_subscription'  :
  priceId === process.env.STRIPE_PRO_PRICE_ID   ? 'pro_subscription'   :
  priceId === process.env.STRIPE_PLUS_PRICE_ID  ? 'plus_subscription'  :
  priceId === process.env.STRIPE_PLUS_PRICE_ID_US ? 'plus_subscription' :
  null;
```

Falls Plus nicht abgedeckt ist: ergänzen.

## STEP 2 — Prüfe ob Supabase-Update für plus_subscription existiert

Der Webhook muss nach Erkennung des Plans die User-Tabelle updaten. Prüfe ob
für `plus_subscription` ein Supabase-Update passiert:

```ts
await supabase
  .from('profiles')  // oder wie die Tabelle heißt
  .update({ subscription_type: 'plus_subscription', subscription_active: true })
  .eq('id', userId);
```

Falls der Update-Block nur `beta_subscription` und `pro_subscription` kennt:
`plus_subscription` als weiteren Case ergänzen — identische Logik wie pro_subscription.

## STEP 3 — Verify

```bash
grep -n "plus_subscription\|STRIPE_PLUS_PRICE_ID" \
  $(find app/ -name "*.ts" | xargs grep -l "webhook" 2>/dev/null | grep -v node_modules)
```

→ Beide env vars (EUR + USD) müssen im Webhook auftauchen.
→ plus_subscription muss als Case im Plan-Routing stehen.
→ Supabase-Update muss für plus_subscription ausgeführt werden.
