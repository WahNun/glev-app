# Diagnose-Prompt: Einflussfaktoren-Log Pattern

## Kontext

Ich entwickle eine T1D Diabetes-App (Next.js App Router + Supabase + Capacitor iOS) namens **Glev**. Ich möchte einen neuen Log-Typ **„Einflussfaktoren"** (Alkohol, Cannabis, Medikamente, Sonstiges) hinzufügen. Bevor ich mit der Implementierung beginne, brauche ich eine genaue Diagnose, wie die **bestehenden Log-Typen** strukturiert sind — damit der neue Typ das gleiche Pattern konsistent weiterführt.

---

## Aufgabe

Untersuche die folgenden bestehenden Log-Typen in der Codebase und dokumentiere ihr Pattern jeweils in DB-Schema, API/Server-Actions, und UI-Komponenten:

1. **Mahlzeiten-Log** (meal / food / carbs)
2. **Insulin-Log** (insulin / bolus / basal)
3. **Sport-Log** (activity / exercise)
4. **Zyklus-Log** (cycle / menstrual)
5. **Symptome-Log** (symptoms)

---

## Was ich für jeden Log-Typ wissen möchte

### 1. Supabase DB-Schema

- Wie heißt die Tabelle? (z. B. `meal_logs`, `insulin_entries`, …)
- Welche Felder hat sie? (id, user_id, timestamp, type-Enums, Freitext-Felder, Mengenfelder, …)
- Gibt es eine gemeinsame Basis-Tabelle oder sind alle Tabellen unabhängig?
- Wie ist RLS konfiguriert? (Policy-Namen und Bedingungen)
- Gibt es Enums oder separate Lookup-Tabellen für Log-Typen?
- Foreign Keys, Indizes?

### 2. API / Server Actions

- Wo liegen die Server Actions oder API Routes für CRUD dieser Logs?
  (z. B. `app/actions/meal.ts`, `app/api/insulin/route.ts`)
- Wie sieht eine typische `create`-Action aus? (Parameter, Validierung, Supabase-Insert)
- Wird Zod oder ein anderes Schema-Validator verwendet?
- Wie wird der `user_id` gesetzt — aus Session, Auth-Helper, oder manuell?
- Gibt es eine gemeinsame Helper-Funktion für alle Log-Typen?

### 3. UI-Komponenten

- Wo liegt die Log-Eingabe-Komponente für jeden Typ?
  (z. B. `components/logs/MealLogForm.tsx`)
- Welches UI-Pattern wird verwendet: Modal, Bottom Sheet, Inline-Form, eigene Page?
- Wie werden Dropdowns/Selects für Typ-Auswahl implementiert? (shadcn Select, native, …)
- Wie wird der Zeitstempel eingegeben? (DateTimePicker, manuell, Default "jetzt"?)
- Wie wird das Formular abgesendet? (React Hook Form, useState, Server Action direkt?)
- Gibt es eine gemeinsame `LogEntryCard`-Komponente für die Anzeige im Feed?

### 4. Feed / Timeline

- Wie werden alle Log-Typen gemeinsam im Feed angezeigt?
- Gibt es eine gemeinsame Query, die alle Typen zusammenführt?
- Wie werden verschiedene Log-Typen visuell unterschieden (Icons, Farben)?
- Welche Felder werden in der Feed-Karte angezeigt?

### 5. Typen & Interfaces

- Gibt es eine zentrale `types/`-Datei oder `db.types.ts` mit TypeScript-Interfaces für die Logs?
- Wie heißen die Interfaces? (z. B. `MealLog`, `InsulinEntry`, …)
- Gibt es ein Union-Type oder gemeinsames `BaseLog`-Interface?

---

## Ausgabe-Format

Bitte dokumentiere deine Findings strukturiert so:

```
## [Log-Typ Name]

### DB
- Tabelle: `tabellenname`
- Felder: id (uuid), user_id (uuid, FK), logged_at (timestamptz), ...
- RLS: [Policy-Beschreibung]

### API/Actions
- Datei: `pfad/zur/action.ts`
- Create-Signatur: createXxx(data: XxxInput): Promise<XxxLog>
- Validierung: [Zod-Schema oder keins]

### UI
- Formular: `pfad/zur/Komponente.tsx`
- Pattern: [Modal / Bottom Sheet / Page]
- Zeitstempel-Input: [wie implementiert]
- Submit: [wie implementiert]

### Feed
- Query: [wie eingebunden]
- Karten-Komponente: [Pfadname]
```

---

## Abschliessende Empfehlung

Nachdem du alle bestehenden Log-Typen analysiert hast, gib eine **konkrete Empfehlung**:

1. **Tabellenname** für den neuen Einflussfaktoren-Log (passend zur Namenskonvention)
2. **Felder** die die Tabelle haben sollte:
   - `id`, `user_id`, `logged_at` (Standard)
   - `influence_type` (Enum oder Text: `alcohol | cannabis | medication | other`)
   - `details` (Text, optional)
   - `amount` (Text oder Numeric, optional)
3. **Dateinamen** für Server Action und UI-Komponente (passend zur Konvention)
4. **Welche bestehende Komponente** als Template am besten geeignet ist (copy-paste Basis)
5. **Was ich bei RLS** beachten muss (copy from existing policy pattern)

---

## Wichtige Hinweise

- **App Store Compliance**: Der Log-Typ heißt in der UI **„Einflussfaktoren"**, nicht „Drogen" oder „Substances". Keine spezifischen illegalen Substanzen in UI-Strings, Enum-Labels oder DB-Kommentaren nennen.
- **Freitext reicht**: Das `details`-Feld erlaubt freie Eingabe — kein vordefiniertes Enum für Substanztypen nötig.
- **Pattern-Konsistenz** hat Priorität: Lieber eine bestehende Konvention übernehmen als etwas Neues erfinden.
