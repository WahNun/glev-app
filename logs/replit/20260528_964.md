# Task #964 — Catch errors in remaining admin actions that threw raw errors

## Summary

Continued the error-handling hardening from Task #961. Four admin server actions in `app/admin/users/actions.ts` were still throwing bare `new Error(...)` instead of redirecting with a visible message. All are now fixed.

## Changes

**`app/admin/users/actions.ts`** — 6 throw sites replaced with redirect calls:

- `confirmEmailAction`: userId guard → `redirect /admin/users?err=action_failed&msg=...`; `auth.admin.updateUserById` failure → `redirect /admin/users/${userId}?err=action_failed&msg=...`
- `restoreUserAction`: userId guard → `redirect /admin/users`; `auth.admin.updateUserById` (ban_duration:"none") failure → `redirect /admin/users/${userId}`
- `sendPasswordResetAction`: userId+email guard → redirect to user detail page if userId is known, else `/admin/users`; `auth.admin.generateLink` failure → `redirect /admin/users/${userId}`
- `sendMagicLinkAction`: same userId+email guard fallback logic; `auth.admin.generateLink` failure → `redirect /admin/users/${userId}`

No UI changes needed — the generic red error banner added in Task #961 at `app/admin/users/[id]/page.tsx` already handles display of `?err=action_failed`.
