## Was wurde gebaut

### 1. Route-Refactoring
- `app/api/ai/confirm-action/route.ts` — `handleConfirmPost(sb, userId, token)` als exportierte Funktion extrahiert (selbes Muster wie `handleInsulinPost`); `POST`-Handler delegiert daran. Ermöglicht direkten Unit-Test ohne Next.js Request-Plumbing.

### 2. Unit-Tests (16 Tests)
- `tests/unit/confirmActionTimelineCheck.test.ts` — vollständige Mock-Supabase-Abdeckung: Happy Path, Upsert-Update, pre-check_type, alle Fehler-Cases (404, 409, 410, 403, Race-Claim, 500+Rollback).

### 3. E2E-Tests (5 Tests gegen Live-Supabase)
- `tests/e2e/confirm-action-timeline-check.spec.ts`:
  1. Happy path — HTTP 200, `ok: true`, UUID `insertedId`, `scheduleReminder` mit korrektem `mealId`/`checkType`/`plannedAt`/`title`/`body`
  2. DB-Persistenz — `meal_timeline_checks`-Zeile hat korrektes `meal_id`, `check_type`, `planned_at` (timestamp-normalisiert), `confirmed_at` (zwischen before/after), `user_id`
  3. Idempotenz-Guard — zweiter Aufruf mit gleichem Token → 409, kein Duplikat in DB
  4. Fehlendes Token → 400
  5. Unbekanntes Token → 404

  Seeding via Admin-Client (service_role), HTTP via `page.request.post()` (Browser-Cookie-Auth, selbes Muster wie `last-appointment.spec.ts`).

### 4. DB-Migration
- `supabase/migrations/20260525_ai_pending_actions_add_timeline_check_kind.sql` — schließt Lücke aus Task #691: `add_timeline_check` fehlte in der `kind`-CHECK-Constraint von `ai_pending_actions`. DROP + ADD CONSTRAINT, additiv. Via Supabase Management API auf Dev-DB applied.

## Bekannte Stolpersteine

- `ai_pending_actions.token` ist UUID-Typ → Test-Token muss `randomUUID()` sein, kein freier String.
- Supabase gibt `planned_at` als `+00:00` zurück, nicht als `Z` → Vergleich via `new Date().getTime()`.
- Die 59 s beim ersten Test-Lauf waren Next.js Turbopack-Kalt-Kompilierung, kein echter Fehler.
