# Fix Report — Task 1154
## Inactive Meal Chips: ✕ Button ohne Blockade durch pointerEvents

**Datum:** 2026-06-04

## Was wurde gemacht

### Problem
Inaktive Mahlzeit-Chips (spätere Einträge im selben KI-Turn) hatten `pointerEvents: "none"` am Container-Div gesetzt. Das hat den ✕-Button unerreichbar gemacht — Nutzer konnten einen unerwünschten Chip nicht überspringen, ohne zuerst den vorherigen zu lösen.

### Lösung

**`components/GlevAIChatSheet.tsx`:**
- `pointerEvents: inactive ? "none" : "auto"` vom Mahlzeit-Chip-Container entfernt. Der Container blockiert keine Pointer-Events mehr global.
- „Engine öffnen →"-Button behält `disabled={busy || inactive}` — natives HTML-`disabled` unterdrückt bereits Pointer-Events auf diesem Button, kein zusätzliches CSS nötig.
- ✕-Button: `disabled`-Bedingung von `disabled={busy || inactive}` auf `disabled={busy}` eingeschränkt — er feuert jetzt auch auf inaktiven Chips.
- Ghost-Styling für ✕ bei inaktivem Zustand hinzugefügt: `background: var(--border-soft)`, `border: 1px solid var(--border)`, `borderRadius: 4` — visuell unterscheidbar vom unsichtbaren Ghost des aktiven Chips.
- Badge-Count (`Mahlzeit N von M`) aktualisiert sich automatisch beim Dismiss: `cancelAction` setzt den State → "cancelled", `totalUnresolvedMeals` zählt nur `pending`/`confirming`, der dismissed Chip fällt beim nächsten Render aus dem Count heraus.

**`tests/unit/mealChipGating.test.ts`:**
- Veralteten Test „inactive meal chip hat pointerEvents: 'none'" → neu: Container-Style enthält NICHT mehr `pointerEvents: inactive`.
- Veralteten Test „non-meal chip behält Bestätigen-Button" → prüft jetzt „Schnell speichern" (Task #1158 hatte es ersetzt).
- Neu: „inactive ‚Engine öffnen →'-Button ist disabled wenn inaktiv" (anchored auf `onClick={onOpenEngine}`).
- Neu: „inactive ✕-Dismiss-Button ist NICHT disabled wenn inaktiv (nur wenn busy)".
- Doku-Kommentar aktualisiert.
